# Agent 产品调研报告

> 调研时间：2026-07-05
> 数据来源：Anthropic 官方指南、Lilian Weng Agent 综述、arxiv 2404.11584 调研论文

---

## 一、市场背景与趋势

### 1.1 Agent 为什么现在爆发

- LLM 能力到达拐点：推理、代码生成、工具调用能力已足够支撑复杂任务
- 成本快速下降：GPT-4o、Claude 3.5 等模型成本较一年前下降 80%+
- 工具生态成熟：Function Calling、MCP 协议、LangChain 等基础设施完善
- 用户预期变化：从「问答」转向「帮我做完一件事」

### 1.2 市场分层

| 层级 | 定位 | 代表产品 | 目标用户 |
|------|------|---------|---------|
| **通用 Agent 平台** | 多场景、可定制 | AutoGPT、LangGraph、CrewAI | 开发者 |
| **垂直场景 Agent** | 深耕单一领域 | Cursor（代码）、Sierra（客服）、Devin（软件工程） | 行业用户 |
| **Agent 开发框架** | 帮别人建 Agent | LangChain、AutoGen、OpenAI Assistants API | 开发者/企业 |
| **Agent 即服务** | 开箱即用 | ChatGPT（with tools）、Claude（with tools） | 大众用户 |

## 二、Agent 核心架构

### 2.1 三大核心组件（来自 Lilian Weng 综述）

```
┌─────────────────────────────────────────┐
│              Agent 系统                  │
│                                         │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
│  │ Planning │  │ Memory  │  │ Tool    │ │
│  │ 规划     │  │ 记忆    │  │ Use 工具 │ │
│  │         │  │         │  │         │ │
│  │ - 任务拆解│  │ - 短期  │  │ - API   │ │
│  │ - 反思   │  │ - 长期  │  │ - 搜索  │ │
│  │ - 自纠正 │  │ - 检索  │  │ - 代码  │ │
│  └─────────┘  └─────────┘  └─────────┘ │
│                                         │
│          LLM (核心控制器)               │
└─────────────────────────────────────────┘
```

### 2.2 Anthropic 的务实建议：从简单到复杂

Anthropic 提出了 5 层渐进式架构，核心原则是**「不要一开始就上最复杂的」**：

| 层级 | 模式 | 描述 | 适用场景 |
|------|------|------|---------|
| L1 | 单次 LLM 调用 | 输入→输出，无工具 | 翻译、摘要、分类 |
| L2 | LLM + 检索 (RAG) | 检索外部知识辅助回答 | 知识库问答 |
| L3 | LLM + 工具调用 | 单步工具使用 | 查天气、查库存 |
| L4 | Workflow（工作流） | 预定义多步流程，每步用 LLM | 文档处理流水线 |
| L5 | Agent（自主） | LLM 自主决定下一步做什么 | 开放式任务 |

**关键洞察**：Anthropic 明确指出，大多数生产场景 L3-L4 就够了，L5 自主 Agent 要慎用——自主性越高，可控性越低，成本越高。

### 2.3 单 Agent vs 多 Agent（来自 arxiv 调研论文）

| 架构 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **单 Agent** | 简单、可控、成本低 | 复杂任务上下文爆炸 | 中等复杂度任务 |
| **多 Agent 协作** | 分工明确、并行、可扩展 | 通信开销、协调复杂 | 大型复杂任务 |

多 Agent 的关键设计决策：
- **领导模式**：有无 leader 统筹？leader 的权限范围？
- **通信方式**：直接对话 vs 共享黑板 vs 消息队列
- **角色分工**：固定角色 vs 动态分配

## 三、Agent 产品的核心能力维度

### 3.1 用户视角：Agent 应该做好什么

| 能力 | 用户期望 | 产品要求 |
|------|---------|---------|
| **理解意图** | 说模糊的话也能懂 | 意图识别 + 上下文补全 + 主动澄清 |
| **任务规划** | 自动拆解，不用我教 | 任务分解 + 依赖管理 + 进度可视化 |
| **工具使用** | 能连我的系统 | 丰富的工具集成 + 自定义工具支持 |
| **记忆能力** | 记住我说过的话 | 短期上下文 + 长期向量记忆 + 用户画像 |
| **错误处理** | 出错了能自己修 | 自我反思 + 重试 + 优雅降级 |
| **透明可控** | 知道它在干什么 | 执行过程可视化 + 人工干预 + 撤销 |
| **结果交付** | 给我能直接用的东西 | 结构化输出 + 多格式 + 可导出 |

### 3.2 产品视角：差异化竞争点

1. **垂直深度 > 通用广度**：深耕一个场景做透，比覆盖 10 个场景做浅更有竞争力
2. **可控性 > 自主性**：用户要的是「可控的强大」，不是「失控的智能」
3. **可观测性是信任基础**：用户看得见 Agent 在做什么，才会信任它
4. **成本效率决定留存**：Agent 比 Chat 贵 10-100x，成本不控住用户会流失
5. **集成生态是护城河**：能连接用户已有的工具链，迁移成本就是护城河

## 四、竞品分析

### 4.1 通用型 Agent

| 产品 | 定位 | 核心能力 | 优势 | 不足 |
|------|------|---------|------|------|
| ChatGPT (GPT-4o) | 通用助手 + 工具 | 对话、代码、搜索、图像 | 用户量大、生态强 | Agent 能力偏弱，自主性有限 |
| Claude (Anthropic) | 通用助手 + 代码 | 长上下文、代码、分析 | 上下文窗口大、安全性好 | 工具集成少于 ChatGPT |
| Gemini (Google) | 多模态助手 | 搜索、图像、Workspace 集成 | Google 生态 | Agent 能力成熟度低 |

### 4.2 垂直型 Agent

| 产品 | 场景 | 核心能力 | 差异化 |
|------|------|---------|--------|
| Cursor / Windsurf | 代码编辑 | AI 原生 IDE、多文件编辑、Agent 模式 | 深度编辑器集成 |
| Devin / OpenHands | 软件工程 | 端到端开发任务、自主调试 | 全流程自动化 |
| Sierra | 客服 | 对话式客服、工单处理 | 品牌定制 + 质量保证 |
| Lindy / Zapier Central | 自动化办公 | 邮件、日历、文档自动化 | 低代码 + 集成丰富 |
| Perplexity | 搜索 | 多步搜索 + 引用 | 研究型 Agent |

### 4.3 Agent 开发框架

| 框架 | 定位 | 优势 | 劣势 |
|------|------|------|------|
| LangChain / LangGraph | 全栈 Agent 框架 | 生态最大、功能全 | 复杂度高、抽象泄漏 |
| CrewAI | 多 Agent 协作 | 简单易用、角色驱动 | 功能较少 |
| AutoGen (Microsoft) | 多 Agent 对话 | 研究导向、灵活 | 生产化不够 |
| OpenAI Assistants API | 托管 Agent | 无需自建基础设施 | 锁定 OpenAI 生态 |
| Dify / Coze | Agent 应用平台 | 可视化、低代码 | 定制性受限 |

## 五、产品建议：Agent 应该做成什么样

### 5.1 核心原则

1. **从 Workflow 起步，逐步走向 Agent**：先用固定流程验证价值，再增加自主性
2. **人在回路（Human-in-the-loop）**：关键决策点必须有人工审核
3. **可观测优先**：每一步执行都可见、可审计、可回溯
4. **成本透明**：让用户知道每个任务花了多少 token / 费用
5. **优雅降级**：Agent 做不了的事，明确告知，不要硬撑

### 5.2 MVP 定义

**第一版只做一件事，做到极致：**

选择一个高频、痛点明确、可量化结果的场景：
- 输入：用户的一句话指令
- 过程：Agent 自主规划 + 执行 + 反馈
- 输出：可直接使用的结构化结果
- 保障：全程可视化 + 可中断 + 可回退

### 5.3 能力优先级（RICE 评分）

| 能力 | Reach | Impact | Confidence | Effort | RICE | 优先级 |
|------|-------|--------|------------|--------|------|--------|
| 意图理解 + 主动澄清 | 10 | 3 | 0.9 | 3 | 9.0 | P0 |
| 工具集成（核心 3-5 个） | 8 | 3 | 0.8 | 4 | 4.8 | P0 |
| 执行过程可视化 | 10 | 2 | 0.9 | 2 | 9.0 | P0 |
| 任务规划与拆解 | 8 | 3 | 0.7 | 5 | 3.4 | P1 |
| 短期记忆 + 上下文管理 | 8 | 2 | 0.8 | 3 | 4.3 | P1 |
| 错误恢复 + 重试 | 6 | 3 | 0.7 | 4 | 3.2 | P1 |
| 长期记忆 + 用户画像 | 5 | 2 | 0.6 | 5 | 1.2 | P2 |
| 多 Agent 协作 | 3 | 2 | 0.5 | 8 | 0.4 | P3 |
| 自定义工具支持 | 3 | 2 | 0.6 | 5 | 0.7 | P2 |

### 5.4 不做什么（Anti-Goals）

- 不做全场景通用 Agent（MVP 阶段聚焦单一场景）
- 不做多 Agent 协作（复杂度过高，收益不明确）
- 不做完全自主无人干预模式（安全风险高）
- 不做自建模型（用 API 即可，聚焦应用层）
- 不做纯聊天机器人（没有工具调用就不是 Agent）

## 六、技术架构建议（产品视角）

### 6.1 分层架构

```
┌─────────────────────────────────────┐
│         用户交互层                   │
│  对话界面 / API / SDK / 插件        │
├─────────────────────────────────────┤
│         Agent 编排层                 │
│  意图识别 → 任务规划 → 执行 → 反思  │
├─────────────────────────────────────┤
│         能力层                       │
│  工具调用 / 记忆 / 检索 / 代码执行   │
├─────────────────────────────────────┤
│         基础设施层                   │
│  LLM API / 向量库 / 任务队列 / 监控  │
└─────────────────────────────────────┘
```

### 6.2 关键产品决策点

| 决策点 | 选项 A | 选项 B | 建议 |
|--------|--------|--------|------|
| 模型选择 | 单一模型 | 多模型路由 | B - 简单任务用小模型，复杂任务用大模型 |
| 记忆方案 | 纯上下文 | 上下文 + 向量库 | B - 长期记忆用向量库 |
| 工具调用 | 固定工具集 | 可扩展工具市场 | A 起步，后续过渡到 B |
| 执行模式 | 同步阻塞 | 异步流式 | B - 流式输出用户体验好 |
| 部署方式 | SaaS | 私有化 | SaaS 起步，按需提供私有化 |

## 七、成功指标定义

### 7.1 北极星指标

**任务完成率（Task Success Rate）**：用户发起的任务中，Agent 成功完成的比例

### 7.2 护栏指标

| 指标 | 定义 | 目标 |
|------|------|------|
| 任务完成率 | 成功完成的任务 / 总任务 | > 70% (MVP) → 90%+ (成熟期) |
| 平均完成步数 | 完成一个任务平均需要多少步 | 越少越好，无绝对目标 |
| 平均任务成本 | 每个任务消耗的 token 费用 | < $0.50 (简单) / < $2 (复杂) |
| 首次成功率 | 无需用户干预一次完成的比例 | > 50% |
| 用户满意度 | 用户对结果的主观评分 | > 4.0 / 5.0 |
| 错误恢复率 | 出错后 Agent 自行恢复的比例 | > 60% |
| 任务放弃率 | 用户中途放弃的比例 | < 20% |

## 八、风险与挑战

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 幻觉/错误执行 | 产出错误结果，用户失去信任 | 结果验证 + 人工审核 + 信心度标注 |
| 成本失控 | Agent 比 Chat 贵 10-100x | 步数限制 + 成本预警 + 小模型路由 |
| 安全/权限 | Agent 误操作用户数据 | 最小权限原则 + 操作确认 + 审计日志 |
| 延迟过高 | 多步执行导致响应慢 | 流式输出 + 异步通知 + 进度反馈 |
| 评测困难 | 难以量化 Agent 质量 | 标准化测试集 + LLM-as-Judge + 人工抽检 |
