# Agent-Smith `shell/` 模块审计报告

## 摘要

本轮对抗式验证共确认 **12 条真实缺陷**（原始 13 条，其中 2 条按同一根因合并为 1 条）。全部经代码路径逐条核实、confidence 均为 high（仅 `oneLine` 一处为 medium），无 P0，**P1 4 条、P2 8 条**。

**按严重度**：P0 = 0，P1 = 4，P2 = 8

**按维度**：
- `state-streaming-concurrency`（状态/流式/并发）= 4
- `terminal-rendering`（终端渲染）= 5
- `correctness`（正确性）= 3

**总体评价**：风险最集中于 **SSE 传输与流式状态管理**——SSE 的 CRLF 跨块归一化缺陷（P1）会把已成功完成的一次 run 误报为错误并污染转录，是本轮最危险的问题；技能域门禁重试会残留永久"进行中"幽灵块（P1）同属流式状态收敛缺陷。其次是**终端渲染层**：多处对 UTF-16 code unit 直接 `slice`/`length` 计算而不感知字素簇与显示列宽，以及 `<Static>` 卸载重挂导致的整段转录重复打印（P1）。`correctness` 类的三条（代码围栏误判、Down 键清空草稿、图片路径基准错配）均为可稳定复现但影响面较窄的功能缺陷。整体没有崩溃或持久数据损坏，但 SSE 与幽灵块两条 P1 直接损害当前"终端工作台体验"这一核心优先级。

## P1 缺陷

### [P1] SSE CRLF 行结束符跨 read 分块被逐块归一化，产生幻影事件边界，导致事件名丢失 / 成功 run 被误报为失败
- **位置**：`shell/src/api.ts:741`
- **维度**：state-streaming-concurrency
- **问题**：`readSseEvents` 对每个 `reader.read()` 解码出的字符串单独执行 `.replace(/\r\n?/g, "\n")` 再追加进 buffer。服务端（sse-starlette 3.4.5，`DEFAULT_SEPARATOR = "\r\n"`）每帧都用 CRLF 分隔。当某个 `\r\n` 恰好被 TCP read 边界切开（chunk A 以 `\r` 结尾、chunk B 以 `\n` 开头）时：A 中孤立的 `\r` 因 `/\r\n?/g` 未匹配到后续 `\n` 被替换成 `\n`，B 的 `\n` 保留，拼接后单个 `\r\n` 变成 `\n\n`——一个幻影帧边界。根因是把行结束符归一化放在了逐块处理而非累积 buffer 上。
- **失败场景**：服务端发送 `event: done\r\ndata:{...}\r\n\r\n`，其中 `event: done` 后的 `\r\n` 被切开 → `splitSseBuffer` 在 `event: done` 后切出无 data 行的独立块（`parseSseChunk` 返回 null 被丢弃），`data:{...}` 成为独立块并以默认 `eventName="message"` 解析 → `done` 事件名丢失 → `sawDone` 永不置真 → 抛出 `"SSE stream ended before a done event was received"`。**一次已成功完成的 run 对用户显示为错误**，终止状态/reason 丢失。对 `provisional_text_delta` 帧发生同样切分时增量文本被当作永久 message 提交，污染并可能重复渲染转录；`tool_call`/`thinking`/`token_usage`/`approval_required` 则整帧丢失。
- **建议修复**：不要逐块 `replace`：将原始解码字符串直接累积进 buffer，把 CRLF 归一化推迟到 buffer 层（`splitSseBuffer` 之前对整个 buffer 归一化，或直接以 `/\r?\n\r?\n/` 匹配帧边界），确保跨块的 `\r\n` 被当作单个换行处理。
- **验证**：confidence high。依赖版本（sse-starlette 3.4.5 已锁定于 uv.lock）与分隔符判断准确；`\r`/`\n` 为单字节 ASCII，`TextDecoder(stream:true)` 不会跨 read 缓冲，read 边界完全可能落在两者之间。等价脚本实测确认后果链成立。触发依赖 read 边界恰好落在 `\r|\n` 之间，属间歇性（单帧概率低），但多帧/多会话累积可达，触发时影响严重，P1 恰当。

### [P1] 技能域门禁重试会残留一个永远停在 retry 态的幽灵技能块（渲染成持续转圈的活动步骤）
- **位置**：`shell/src/transcript-state.ts:478`
- **维度**：state-streaming-concurrency
- **问题**：`applyStreamEvent` 的 `skill` 分支在 `status==="start"` 时无条件 push 一个新的 running 块（478-491 行），从不复用同名未终结块；而非 start 事件（494-513 行）只匹配 `state==="running"` 的块。后端 `pipeline.py` 域门禁重试流程先发 `SKILL_END status="retry"`（把块置为 retry 态），随后因 `node_idx` 未自增、`while` 回到顶部又重发一次 `SKILL_START`。于是同一技能节点在 transcript 中产生两个块：第一个卡在 retry 态且再不被后续事件命中，第二个才承接 `passed→done`。`transcript.tsx:498` 将 `retry` 与 `running` 同样判为 active，渲染为 ◉ 活动标记，即使整轮 run 早已 done，历史 transcript 里仍永久显示一个"进行中"的技能步骤；多次重试累积多个幽灵块。
- **失败场景**：用户消息触发带域门禁的技能，第一次未过门禁（gate verdict=retry），第二次通过。事件序列 `start→retry→start→passed` 抵达 shell → transcript 永久渲染两个同名技能块：第一个 ◉ 卡在 retry 态像仍在运行，第二个 ● done。整轮完成后转圈幽灵块不消失，误导用户以为还有步骤没跑完。
- **建议修复**：在 `start` 处理里先查找同名且处于非终结态（running/retry）的现有块并复用（重新置为 running），而非无条件 append；或让非 start 分支的匹配放宽到 `running|retry`。二选一即可保证一个技能节点在 transcript 中只对应一个块并收敛到终态。
- **验证**：confidence high。前后端代码路径完整可达且无守卫：`guard.record` 首次失败返回 retry（`max_same=2, attempts=1<2`），SSE 经 `session_service.py:637` 与 `api.ts:669` 映射，前端确实收到 `start→retry→start→passed`。测试仅覆盖单次 retry，未防护此序列。唯一描述瑕疵：块 A 通常已积累 activities，渲染为 ◉ + 陈旧 activities 而非始终 "waiting for the first action…"，但核心缺陷（永久活动幽灵块）成立。

### [P1] 打开 /tokens 或 /runs 面板后返回聊天会重复打印整段 transcript
- **位置**：`shell/src/index.tsx:1108`
- **维度**：terminal-rendering
- **问题**：主渲染树用 `panel !== "tokens" && panel !== "runs" ? (<Static …>) : null` 条件渲染 `<Static>`。进入 tokens/runs 面板时 `<Static>` 从树中卸载，但它此前写入终端的 scrollback 永久留存（Ink 不擦除 static 输出）。返回聊天时（`input.ts:330-332` 的 Esc 仅 `set({ panel: "chat" })`，不调 `clearTerminal`、不 bump `transcriptEpoch`），`<Static>` 以全新实例重挂、内部渲染索引归零，于是把 hero + 所有已完成 entry 再全量写一遍，叠加在旧副本之下。代码库在 `/clear`、`/new`、`/resume` 路径都特意先 `clearTerminal()` 再 bump epoch，正是知道重挂会重印——tokens/runs 卸载/重挂路径漏掉了这层保护。skills/mcp/hooks/sessions 面板不隐藏 `<Static>`，故不受影响，进一步印证是 tokens/runs 独有缺陷。
- **失败场景**：正常聊天数轮 → 输入 `/tokens`（或触发 `panel="runs"`）→ 按 Esc 返回 → 整段对话被完整重打印一遍，scrollback 出现两份 transcript；每 toggle 一次多一份（开关 3 次 = 4 份）。
- **建议修复**：进入或离开 tokens/runs 面板时按 `/clear` 既有模式处理：调用 `clearTerminal()` 并 bump `transcriptEpoch` 让 `<Static>` 干净重挂；或改为始终保持 `<Static>` 挂载（不因面板条件卸载它），让面板叠加在动态区渲染。
- **验证**：confidence high。经 ink 7.1.0 源码证实：`Static.js` 的 `useState(0)` index 在卸载时销毁、重挂归零 → 渲染全部 item；`ink.js` 在 `hasStaticOutput` 时无条件 `stdout.write`，已写入 scrollback 不被擦除。对照 bridge.ts 的既有 `clearTerminal` + bump epoch 保护，正说明开发者知晓重挂会重印。P1 恰当：可重复的终端渲染污染，直接损害核心终端体验，但非崩溃/数据丢失。

### [P1] 含空格 info string 的代码围栏被误判，导致后续内容被吞进幻影代码块
- **位置**：`shell/src/mermaid.ts:18`
- **维度**：correctness
- **问题**：`splitMarkdownBlocks` 用的 `FENCE_PATTERN = /^( {0,3})(`{3,}|~{3,})\s*([^\s`~]+)?\s*$/` 要求开栏行 info string 只能是单个无空格 token 后立即行尾。但 CommonMark 合法允许 info string 含空格（language 只取第一词），如 ` ```ts twoslash ` 或 ` ```python title="a.py" `。此处开栏行因带空格不匹配，不被识别为围栏开始；更糟的是紧随其后的裸闭栏 ` ``` ` 反而匹配 `FENCE_PATTERN`（被当作新开栏，language=text），导致此后内容一路被吞进幻影代码块。同仓库另外两个 splitter（`streaming-markdown.ts:22`、`markdown-layout.ts:6`）用宽松模式 `/^( {0,3})(`{3,}|~{3,})/`，三处不一致印证是本文件疏漏。
- **失败场景**：助手输出含 ` ```ts twoslash\nconst x = 1;\n```\nDone.`——`splitMarkdownBlocks` 无法识别开栏 → 该行与代码被当散文渲染；裸 ` ``` ` 被误判为新围栏开始 → `Done.` 及其后全部内容被吞入代码块正文，整条消息后半段渲染错乱/丢失格式。
- **建议修复**：把 language 捕获改为宽松，仅匹配开栏标记与其后第一个 token 作为语言、允许剩余任意 info string（如 `/^( {0,3})(`{3,}|~{3,})\s*([^\s`~]*)/`），与 `streaming-markdown` / `markdown-layout` 保持一致。
- **验证**：confidence high。多块场景（两真实代码块夹散文）已精确复现，与 finding 所述一致。子场景校正：单块尾随（` ```ts twoslash...```\nDone.`）因 EOF 兜底会把未闭合幻影栏退回 markdown，splitter 内并非"永久吞入"，但退回后经下游 `MarkdownContent`（`transcript.tsx:155`）渲染时 `Done.` 仍被当代码块，用户可观测的渲染污染仍成立。触发依赖 LLM 输出带空格 info string，属合法且中等频率场景。

## P2 缺陷

### [P2] Ctrl+O 视图切换会静默删除输入框末尾的字母 "o"（双重去除）
- **位置**：`shell/src/index.tsx:1022`
- **维度**：state-streaming-concurrency
- **问题**：存在两套互相冲突、都试图移除 Ctrl+O 触发时 TextInput 顺带插入的那个 "o" 的机制：(1) `handleInputChange` 的 `suppressRef` 检查（1019-1023）——当 `onChange` 送来 `${inputValue}o` 时提前 return，不写进 store；(2) `input.ts:92-95` `handleViewToggle` 的 `queueMicrotask` 无条件把 store 里末尾 "o" 用 `slice(0,-1)` 砍掉。派发顺序为 `useShellInput` 先、TextInput 后：`handleViewToggle` 先设 `suppressRef="o"` 并排 microtask；`onChange("fooo")` 命中 suppressRef 被提前 return（store 保持 "foo"，已成功抑制一次）；microtask 再执行发现 "foo" 以 "o" 结尾又砍一刀 → "fo"，合法字符被多删一次。
- **失败场景**：输入以字母 o 结尾的文本（如 "repo"），按 Ctrl+O 切换视图 → 视图切换但末尾 "o" 被静默删除（"repo"→"rep"），无任何提示。非以 "o" 结尾的输入不受影响。
- **建议修复**：只保留一套去除机制——要么删掉 `handleInputChange` 的 `suppressRef` 提前 return，只靠 microtask 去尾；要么删掉 `queueMicrotask`，只靠 suppressRef 抑制。二者不能并存。
- **验证**：confidence high，严重度由 P1 降为 P2。派发顺序比 finding 论证更牢固：ink 7.1.0 `useInput` 用 `useEffectEvent` + 订阅 deps 不含 handler，不会每 render 重订阅；`useShellInput` 在 boot 期挂载、options 恒无 `isActive` 变化 → 永远排 emitter 第一位先派发。双重去除成立、可复现。降级理由：触发面窄（草稿末尾为 "o" + Ctrl+O），仅影响未提交草稿，单字符、非破坏、无崩溃，属轻微隐性输入损坏。

### [P2] spawn 失败（uv 缺失/不可执行）被误报为 30 秒超时，已捕获的真实错误从不被使用
- **位置**：`shell/src/dev-server.ts:233`
- **维度**：state-streaming-concurrency
- **问题**：`launchLocalServer` 通过 `child.once("error")` 捕获 spawn 错误存入 `spawnError`，但 `waitForCompatibleServer` 只在 `launch.child.exitCode !== null` 分支里读取 `getSpawnError()`。当 spawn 因 ENOENT（uv 未安装）或 EACCES 失败时，Node 只触发 `error` 事件、进程从未启动，`exit` 事件在 macOS/Linux 上不触发，`child.exitCode` 恒为 null，该分支永进不去，循环持续做健康探测直到撞上顶层超时才退出。
- **失败场景**：环境未安装 uv → 运行 `ensureLocalServer` → child 触发 `error`(ENOENT)、exitCode 保持 null → `waitForCompatibleServer` 空转 → 抛出误导性的 `"Timed out while starting the local Smith server."`，真正原因 `spawn uv ENOENT` 虽已存入 `spawnError` 却从不展示，严重拖慢首次启动并误导排障。
- **建议修复**：在循环条件中同时检查 `launch.getSpawnError()`：若已存在则立即抛出 `"Could not launch the local Smith server: <message>"`，不必等 exitCode 变化；或在 `child 'error'` 时置 failed 标志并每轮检测。
- **验证**：confidence high。证据链完整，唯一消费点被 `if (launch.child.exitCode !== null)` 守卫，ENOENT 下永不进入。两处细节校正不改变结论：连接被拒是即时 reject 非等满 3s；实际约 20s（40×500ms）迭代耗尽先于 30s 顶层 break 触发。属诊断/错误上报缺陷，P2 合适。

### [P2] `truncate`/`oneLine` 用 String.length/slice 截断，会切断 emoji 代理对与 ZWJ 序列并低估宽字符列宽（合并两处同根因）
- **位置**：`shell/src/index.tsx:94`（`truncate`）；`shell/src/approval.ts:31`（`oneLine`）
- **维度**：terminal-rendering
- **问题**：两处都按 UTF-16 code unit 计长（`text.length`）与切分（`slice(0, max-1)`），不感知字素簇/显示列宽。对含 emoji/星标/CJK 扩展 B 等 astral 字符或 ZWJ 组合序列的文本，slice 若落在代理对或组合序列中间会留下孤立高位代理，终端渲染为替换符 `�`；对 CJK 全角字符，`length` 只按 code unit 计而非显示列宽，被截后实际显示列数远超预期（80 个 CJK 约 160 列），窄终端溢出错位。项目其它处（`mermaid.ts`/`text-layout.ts`）已用 `string-width@8`，唯这两个本地函数未用。`truncate` 被 SessionsPanel、McpPanel、SlashMenuRow、SkillMentionMenu、QueuePreview 等大量调用；`oneLine` 被 approvalDetails 以默认 `maxLength=240` 调用（value 来自 approval.arguments 等 agent/用户可控字符串）。
- **失败场景**：(1) 会话标题/技能描述/排队消息中 emoji 恰好跨越截断边界 → 终端显示 `…�…` 破碎字形；(2) 标题为大量中文时被截到 max 个 code unit，实际占约 2×max 列，窄终端溢出破坏 Box 布局；(3) 审批 detail 的 value >240 字符且第 239/240 位是 emoji 代理对 → `slice(0,239)` 截出孤立高代理项，审批面板出现乱码。
- **建议修复**：改用 code point 或字素簇（`Array.from` / `Intl.Segmenter`）计数与切分，并用 `string-width` 按显示列宽衡量 max，避免切断代理对及低估 CJK 宽度。
- **验证**：`truncate` confidence high、`oneLine` confidence medium。两条失败路径均可达且无上游守卫；`oneLine` 的 medium 因触发需 emoji 恰落在 238/239 边界这一罕见对齐。均为渲染/布局层装饰性瑕疵（最坏 `�…`），不影响审批逻辑或安全决策，P2 恰当。

### [P2] RollingBarChart 强制每列最小宽度 6，窄终端（<46 列）下柱状图溢出换行错位
- **位置**：`shell/src/token-panel.tsx:63`
- **维度**：terminal-rendering
- **问题**：`width = Math.max(6, Math.min(BAR_WIDTH, Math.floor((columns - 4) / days.length)))`，`days.length` 恒为 7。外层 `Math.max(6, …)` 使每列宽度不低于 6，7 列合计恒 ≥ 42。而图表容器可用宽度为 `columns-4`（`index.tsx:1121` 的 `paddingX={2}`），当 `columns < 46` 时 42 > columns-4，7 个 `<Box width={6}>` 在 `flexWrap` 默认 nowrap 行内溢出，日期标签行/数值行与柱体行的列不再对齐。`TokenStatsPanel` 在 `columns<64` 时已切换精简提示，说明窄终端是被预期的使用场景。
- **失败场景**：终端宽 40 列时打开 `/tokens` 的 Stats 标签：7 列各占 6 = 42 > 可用 36 列，柱体、星期标签、token 数三行互相错位/换行，图表不可读。
- **建议修复**：去掉最小宽度 6 的硬下限，或按 columns 动态下调（如 `Math.max(2, Math.min(BAR_WIDTH, Math.floor((columns-4)/days.length)))`），并在极窄终端退化为 OverviewView 那种单行条形展示。
- **验证**：confidence high。临界点精确吻合"<46 列"（45 列可用 41<42 仍溢出）。纯渲染退化，仅影响不常见的 <46 列终端，无崩溃与数据错误，P2 恰当。

### [P2] diff 行折行后，intraline 变更高亮用整行偏移套在首个折行片段上导致加粗错位
- **位置**：`shell/src/diff-block.tsx:244`
- **维度**：terminal-rendering
- **问题**：`changedRanges` 基于完整行文本的字符串偏移计算。`renderDiffLines` 把 `line.text` 按宽度折成多个 fragment，但只把 `changedRanges` 挂到 `index===0` 的 fragment 上（244 行）。随后 `wordSegments(line.content, line.changedRanges)`（331 行）用这些"整行偏移"去 slice 首个折行片段。当变更词落在第二段及以后时，`range.start` 超过首片段长度，`slice` 返回空/截断，变更高亮被丢弃或加粗到错误字符。三段 slice 拼接仍是完整文本（不丢字符），故为渲染缺陷而非数据丢失。
- **失败场景**：宽约 40 的 diff 面板中出现一条被修改的 90 字符长代码行、改动词在第 60 列。折成多段后 `changedRanges` 偏移约 60-65 却只挂在首段（约 0-36 列）→ `slice(60,65)` 起点越界 → 改动词未被加粗；若部分落入首段则加粗到错误字符。
- **建议修复**：折行时把 `changedRanges` 按各 fragment 的起止字符位置切分并重新映射到对应片段（记录每个 fragment 在原文的起始 offset），而非只挂到 index 0；或在渲染端按累计 offset 定位每段应高亮的子区间。
- **验证**：confidence high。代码路径成立且无守卫，触发条件（deletion+addition 配对、修改行长到折行、变更位于首片段外）在真实长代码行 diff 中常见。仅加粗位置错误，不丢字符，P2 恰当。

### [P2] ImageAttachments 以 image.path 作 React key，而 parseImages 不去重，重复路径导致 key 冲突渲染异常
- **位置**：`shell/src/smith-ui.tsx:14`
- **维度**：terminal-rendering
- **问题**：`parseImages`/`parseImage` 只做结构/白名单/长度校验，不按 path 去重；`localProjectImagePath` 用 `path.resolve` 归一化，会把 `"a.png"` 与 `"./a.png"` 归一成同一绝对路径且不校验文件存在。`ImageAttachments` 用 `key={image.path}` 渲染列表，重复 path 即重复 React key，Ink（React reconciler）下触发 duplicate-key 警告并使兄弟节点渲染不确定。
- **失败场景**：模型返回 `images:[{path:"chart.png",alt:"A",width:40},{path:"./chart.png",alt:"B",height:20}]`（或直接两个相同 `"chart.png"`）→ 两者归一为同一 path，校验通过，map 出两个相同 key 的 `<Box>` → 重复 key 警告污染终端，初次挂载节点不确定。
- **建议修复**：key 用稳定唯一值（如列表 index），或在 `parseImages` 中对重复 path 去重/拒绝。
- **验证**：confidence high。触发路径成立、无守卫。两点校正：(1) 无需归一化差异，两个相同字面 path 同样碰撞；(2) smith-ui 是 deliberately static 渲染无 re-render，主要后果是重复 key 警告与初次挂载节点不确定，而非 finding 所述的"reconciliation 合并/丢弃第二张"。需模型输出两条指向同一文件的项（可发生但非常规），P2 恰当。

### [P2] Down 方向键在未从历史检索的状态下会清空正在编辑的输入（静默丢失已输入文本）
- **位置**：`shell/src/input.ts:374`
- **维度**：correctness
- **问题**：`handleHistoryNavigation` 向下分支有缺陷。`historyIndex` 初始为 -1，打字时 `handleInputChange` 从不修改它（提交后 `pushHistory` 也重置为 -1）。向下分支 `const index = state.historyIndex === -1 ? -1 : state.historyIndex + 1;`，当 historyIndex 为 -1 时 index 仍为 -1，`index >= inputHistory.length` 不成立（历史非空），进入 else 分支执行 `state.set({ historyIndex: index, inputValue: index === -1 ? "" : ... })`，把 inputValue 设为空串。ink-text-input 收到 downArrow 会提前 return 不产生 onChange，故 `handleHistoryNavigation` 是 Down 键唯一处理者，行为与按键顺序无关，必然触发。主流 shell（bash/zsh）在已处于最新行且未检索历史时按 Down 不会清空当前行。
- **失败场景**：先发送过一条消息（inputHistory 非空，historyIndex=-1），随后在 chat 面板输入 "important message"（historyIndex 仍 -1），按一次 Down → 命中 else 分支，inputValue 被设为 ""，文本静默清空且无法恢复。
- **建议修复**：向下分支中当 `historyIndex === -1` 时直接 `return false`（Down 为 no-op，不覆盖当前草稿）；仅当 historyIndex 为有效历史下标时才继续向新方向推进。
- **验证**：confidence high，严重度由 P1 降为 P2。触发路径逐条印证，前置 handler 均 return false 使 Down 落到此处，失败场景完全可达且无守卫。降级理由：确为静默输入丢失，但需先有历史记录、仅影响单行 composer 且可重新输入，单次误按代价有限。

### [P2] localProjectImagePath 用 process.cwd() 解析相对路径，却拿 SMITH_PROJECT_CWD 做边界比较，二者不一致时合法图片被静默丢弃
- **位置**：`shell/src/smith-ui-schema.ts:202`
- **维度**：correctness
- **问题**：`projectRoot` 取自 `SMITH_PROJECT_CWD`（或 cwd），而 `resolved = path.resolve(value)` 对相对路径是相对于 `process.cwd()` 解析。当 shell 进程实际工作目录与 `SMITH_PROJECT_CWD` 不同（该环境变量存在的意义正是覆盖 cwd），相对图片路径会被解析到错误基准目录，`!resolved.startsWith(projectRoot)` 判为项目外而返回 null。安全上是 fail-closed（不放行项目外文件），但功能上会丢图。
- **失败场景**：启动时 `SMITH_PROJECT_CWD=/home/u/proj` 而 shell 进程 cwd=`/home/u`。模型返回 `images:[{path:"charts/plot.png",...}]` → `path.resolve("charts/plot.png")` = `/home/u/charts/plot.png`，不以 `/home/u/proj` 开头 → 返回 null → `parseImages` 返回 null → 整个 smith_ui payload 校验失败，回退成 fallback 代码块，结构化 UI 与图片全部不渲染。
- **建议修复**：相对路径解析也用 projectRoot 作基准：`const resolved = path.resolve(projectRoot, value);`
- **验证**：confidence high。`cli_shell.py` 以 `env.setdefault('SMITH_PROJECT_CWD', str(Path.cwd()))` 设项目目录，同时用 `subprocess.run(..., cwd=_repo_root())` 把 node shell 子进程工作目录固定为仓库根——典型用法下二者必然不同，是默认常态而非边界。绝对路径不受影响，仅相对路径被系统性丢弃。维持 P2：功能降级 + 优雅回退，安全 fail-closed，无崩溃或数据损坏。
