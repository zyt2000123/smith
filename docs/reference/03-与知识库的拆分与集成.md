# 03 · 与 AI Knowledge Hub 的拆分与集成

## 拆分原则

**知识是团队的，员工是个人的。**

- 团队知识、文档权限、Wiki 浏览、知识库申请与上传留在 AI Knowledge Hub。
- 数字员工的人格、任务、记忆、本地工具、权限和工作记录留在本机 Agent-Smith。
- 两边通过 MCP 或检索 API 连接；Hub 对 Agent-Smith 来说是一组可调用的团队知识工具。

## 两个产品的边界

| | Agent-Smith | AI Knowledge Hub |
|---|---|---|
| 核心对象 | 数字员工、任务、技能、连接器、权限、记忆 | 知识库、文档、Wiki、用户、权限 |
| 主要用户动作 | 创建员工、派任务、看工作记录、配置能力 | 上传文档、管理知识库、检索和浏览 Wiki |
| 数据归属 | 个人电脑 | 团队服务器 |
| 运行环境 | 本地桌面应用 | 云端 Web 服务 |
| 对外能力 | 调用 MCP / 本地工具 | 提供 MCP server / 检索 API |

## Hub 侧定位

Hub 回归「团队知识基础设施」：

保留并继续投入：

- 文档 ingest 管线
- OKF bundle
- 混合检索
- 知识库权限、申请、上传
- Wiki 浏览前端
- MCP server 和 token 管理

冻结或不扩：

- Hub 内部的多 Agent 雇佣层
- 服务端多 Agent 讨论循环
- 与数字员工管理重叠的 `/workflows`、`/skills` 占位产品化方向

原因：这些能力应该落在本地 Agent-Smith，避免云端 Hub 同时变成知识库和 Agent 平台。

## Agent-Smith 如何使用 Hub

### 1. 员工绑定知识库

员工配置里记录可访问的知识库范围：

```yaml
knowledge:
  provider: hub
  connector: default-hub
  allowed_groups:
    - product
    - engineering
```

任务执行时，检索工具只在员工允许范围内调用。

### 2. 对话任务中检索

当员工需要团队知识时：

1. AgentScope 工具层调用 Hub MCP 或检索 API。
2. Hub 返回答案片段、来源 slug/URL、权限过滤后的结果。
3. Agent-Smith 在消息流或任务结果里展示来源回链。

### 3. 右侧能力面板展示

对话页右侧面板里的「知识库」展示当前任务可用的 Hub 知识范围；「MCP」展示已连接的 Hub MCP server；「权限」展示该任务是否允许调用。

这比把知识库藏在设置页里更符合任务执行心智。

## 集成契约

| 项 | 约定 |
|---|---|
| 首选协议 | MCP over streamable-http |
| 候选协议 | Hub 内部检索 HTTP API |
| Hub MCP 端点 | `https://<hub-host>/mcp/mcp` |
| 鉴权 | `Authorization: Bearer <mcp_token>` |
| 工具发现 | 标准 MCP `listTools()` |
| 来源回链 | 返回 wiki slug/URL |
| 权限语义 | 以 Hub 用户可见范围为上限，Agent-Smith 再做员工级过滤 |
| 故障语义 | Hub 不可用时，本地员工仍可工作，但知识检索能力降级 |

## Hub 侧需要支持的能力

优先级从高到低：

1. token 自助生成页：用户能从 Hub 网页复制给 Agent-Smith。
2. 按知识库 / group 过滤的检索工具：`search_kb(group, query)` 或等价参数。
3. 读全文工具：`read_doc(slug)`。
4. 最新文档工具：`list_recent(group)`。
5. token 绑定用户身份，检索范围自动等于用户可见范围。

## Phase 0 必须核实

1. Hub 现有 MCP server 暴露哪些工具。
2. MCP token 是否能绑定用户可见范围。
3. 现有检索 API 是否支持按 group/知识库过滤。
4. Agent-Smith 内部默认走 MCP 还是直调检索 API。

默认判断：MCP 是标准接缝；如果首版为了速度和过滤精度需要直调 Hub API，也可以先直调，但接口应封装在 `HubConnector` 里，避免后续迁移成本扩散。
