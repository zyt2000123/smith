# 02 · 系统架构

## 核心设计哲学

**宏观确定性控制，微观 LLM 自由度。**

代码（规则引擎、DAG、门禁）管流程，LLM 只在 ReAct 循环里获得「做事」的自由。这避免了纯 LLM Agent 的不可控问题。

运行定位上，Agent-Smith 是 **terminal-native / shell-native** 的本地进程 Agent。`shell` 是一等执行能力；`read_file`、`write_file`、`edit_file`、`grep`、`git_ops`、`web_fetch` 等结构化工具，是对 shell/file/git/web 高频操作的安全封装。安全边界按能力划分（文件、网络、Git 副作用、删除、写入、secret 访问），而不是按工具名粗暴区分。

> 当前实现：Smith 是唯一运行中的 Agent，运行时档案固定为 `~/.agent-smith/agent/`。领域扩展通过启动扫描 `agents/identities/*.yaml` 完成；不保留按“员工”创建档案或运行目录的兼容层。

## 一、核心分层架构

```
shell/ ── HTTP/SSE ──→ server/ ── import ──→ engine/ ── import ──→ common/
                                │                      ↑
                                └──── 读取 ────→ agents/ ┘
```

依赖方向单向不可逆。`common` 在最底层不依赖任何其他层。`agents/` 是纯内容，被 engine 加载。

| 层 | 目录 | 职责 | 语言 | 关键约束 |
|---|---|---|---|---|
| 基础设施 | `common/` | 路径、数据库连接、YAML 工具 | Python | 零业务逻辑，不依赖任何其他层 |
| 执行框架 | `engine/` | 自研 Agent 框架（LLM、执行引擎、记忆、安全） | Python | 不知道 FastAPI、不知道「Agent 实例管理」概念 |
| 内容定义 | `agents/` | Smith 身份、内置技能、工具 provider、安全规则 | .md/.yaml/.json + Python | 能力扩展靠 skill/tool/knowledge，不靠新建 Agent |
| 平台后端 | `server/` | 本机后端（Agent Profile、会话、任务、API） | Python | 薄壳路由 → 服务编排 → Repository 持久化 |
| 终端前端 | `shell/` | Ink/React 终端工作台 | TypeScript | 通过 HTTP/SSE 与 server 通信，不直接调用 engine |

**分层测试**：把 FastAPI 换成 Flask，只需重写 `server/`——engine、agents、common 完全不动。把终端前端换成 Web/macOS 客户端，也只改前端层。

#### 2.2.1 四层执行引擎

用户发一条消息后经历四层处理：

```
用户消息："帮我修复登录页的表单验证 bug"
        │
   ┌────▼──────────────────────────────┐
   │ 第 1 层：任务路由（task_router）     │  代码控制
   │ "修复" + "bug" → Bug Fix 路由      │  规则引擎，不用 LLM
   └────┬──────────────────────────────┘
        │
   ┌────▼──────────────────────────────┐
   │ 第 2 层：技能链 DAG（skill_chain）  │  代码控制
   │ sde-debug → planning → testing    │  有向图，强制顺序
   │ → change-validation → code-review │
   └────┬──────────────────────────────┘
        │ 对每个技能节点
   ┌────▼──────────────────────────────┐
   │ 第 3 层：ReAct 循环（agent_loop）   │  LLM 控制
   │ 思考 → 调工具 → 观察 → 思考 → ... │  这里 LLM 有自由度
   └────┬──────────────────────────────┘
        │ 每次调工具前
   ┌────▼──────────────────────────────┐
   │ 第 4 层：安全护栏（tool_guard）     │  代码控制
   │ 正则匹配 29 条规则，拦截危险操作    │  9 类安全规则
   └───────────────────────────────────┘
```

第 1、2、4 层不用 LLM——「先规划再编码」「修 bug 必须先调试」「不能 rm -rf」是确定性规则。LLM 只在第 3 层获得自由。

**声明式路由**（`engine/identity_catalog.py` + `task_router.py`）：

| 输入特征 | 路由结果 | 处理方式 |
|---|---|---|
| 匹配身份 YAML 的 `examples` / `keywords` | `RouteDecision(identity_id, route_id, pipeline_id)` | 加载身份声明的 pipeline |
| 未命中任何路由 | 默认身份 + `direct` | 跳过 pipeline，直接 ReAct |
| 会话已固定身份 | 在固定身份内匹配路由 | 保持领域上下文与记忆边界 |

**技能链定义**（`skill_chain.py`）：

```
身份 YAML 的 `pipeline` 标识对应 `agents/pipelines/*.yaml` 中声明的 `route`。
节点、门禁和回退边全部由 pipeline YAML 定义；新增法务、财务等领域时新增
YAML 文件，不修改 Python 路由代码。
```

#### 2.2.2 门禁系统（`gate.py`）

双层门禁：每个技能执行后先过 Skill Rubric Gate（通用质量检查），再过节点专属 Gate。

| 类型 | 门禁 | 通过条件 |
|---|---|---|
| 流程门禁 | PlanningGate | 步骤 >= 3 + 有验证点 |
| | RootCauseGate | 有根因声明 + 证据支撑 |
| | DesignGate | 涉及文件 + 数据流 + 依赖关系 |
| | UnderstandingGate | 理解验证通过 |
| | ContractAlignmentGate | 接口契约对齐 |
| 测试门禁 | TestGate | 测试用例 >= 2 + 边界覆盖 |
| | TestDeliveryGate | 测试实际运行 + 有 pass/fail 计数 |
| 验证门禁 | ValidationGate | 有执行证据 + pass/fail 结果 |
| | ReviewGate | 有 P0/P1 分级 + 可操作发现 |
| 通用门禁 | SkillRubricGate | 长度 > 50 + 代码块/文件引用 + 无错误标记 |
| 交付门禁 | GitWorktreeGate | Worktree 就绪 + 无冲突 |
| | PRGate | Conventional Commit 格式 + 无敏感文件 |
| LLM 门禁 | LLMGate | LLM 判断输出质量是否达标 |

共 13 个具体门禁实现。

#### 2.2.3 回退 + Failure Loop Guard（`backtrack.py`）

```
回退路径：
  Implementation 失败 → 回退到 Planning
  Design 失败       → 回退到 Understand
  Verify 失败       → 回退到 Planning

Failure Loop Guard：
  1. 记录失败签名 = hash(error_type + context_hash)
  2. 同一签名出现 2 次 → 换策略（不是简单重试）
  3. 最多 2 种替代策略 → 全部失败 → blocked，请求人工介入
```

#### 2.2.4 System Prompt 10 层组装（`assembler.py`）

| 层 | 来源 | 职责 | 裁剪优先级 |
|---|---|---|---|
| 1 | role.md | 核心使命（最高优先级） | 不裁剪 |
| 2 | style.md | 行为风格 | 6（最后裁剪） |
| 3 | workflow.md | 工作流 SOP + 门禁定义 | 5 |
| 4 | toolbox.md + 工具注册表 | 可用工具声明 + 使用规范 | 4 |
| 5 | 技能注册表 | 可用技能摘要 | 3 |
| 6 | context.md | 审核后的用户协作记忆（带参考围栏） | 不裁剪 |
| 7 | SMITH.md | 用户维护的全局/项目规则 | 不裁剪 |
| 8 | output_style.md | 输出格式规范 | 1（最先裁剪） |
| 9 | memory/ | recent + 按需召回的 durable/episodes | 2 |
| 10 | 运行时上下文 | Agent Profile / session_id / 环境 | 不裁剪 |

超 token 预算时依次裁剪 output style、项目记忆、技能目录、工具目录和 style。role、workflow、context、`SMITH.md` 与运行时上下文保留。稳定层(1-5)的 MD5 哈希用于 LLM prefix cache hint。

记忆层（Layer 8）注入时包裹在安全围栏中，标注为"untrusted historical reference material"以防止 prompt injection。

#### 2.2.5 记忆系统

**当前记忆模型**（详见 `docs/05-Engine-记忆系统.md`）：

```
~/.agent-smith/agent/memory/
├── recent.jsonl        # 已清洗的追加式证据
├── recent.md           # 审核后的近期工作记忆
├── durable.md          # 审核后的长期项目记忆
├── memory_history.jsonl# 编译/审核/写入审计
└── episodes/           # 主题摘要与可丢弃 FTS5 索引
```

`context.md` 位于 profile 根目录。`recent.jsonl` 经 `compile_context()`、`compile_recent()`、`compile_durable()` 分流；每份完整 Markdown 都由 Compiler 生成、Reviewer 审核，再由代码校验并原子写入。唯一规则是 `engine/memory/MEMORY_POLICY.md`。

**Dream**（`dream.py`）：清洗所有记忆层，整理 durable；整理结果同样必须经过 Reviewer。

**Episode 检索**（`search.py`）：SQLite FTS5 trigram + BM25 排序；派生索引损坏时自动重建。

**偏好学习**（`user_learner.py`）：4 维启发式检测，同一特征观察 3 次后产出 evidence signal，不直接写 `context.md`。

#### 2.2.6 安全体系

四层安全：

1. **工具调用层** — 29 条规则，9 类拦截（command_injection, code_execution, destructive_command, network_abuse, platform_integrity, privilege_escalation, resource_abuse, sensitive_file_access, shell_evasion）
2. **文件访问层** — 路径白名单 + 敏感目录黑名单
3. **事实核查层** — `fact_gate.py` 事实验证
4. **交付安全层** — PRGate 禁止敏感文件

## 三、LLM 配置四层覆盖

配置合并逻辑在 `engine/llm/model_config.py` 的 `resolve_llm_config()`。

| 层级 | 位置 | 优先级 |
|---|---|---|
| 环境变量 + 平台级 | `AGENTSMITH_LLM_*` 环境变量 + `~/.agent-smith/config.yaml` | 最低 |
| Smith 出厂级 | `agents/smith/config.yaml` | ↑ |
| Smith 运行时级 | `~/.agent-smith/agent/config.yaml` | ↑ |
| 会话级 | API 参数传入 | 最高 |

从上到下逐层 deep merge（`yaml_utils.merge_configs`），下层覆盖上层已有字段。

支持的环境变量：`AGENTSMITH_LLM_API_KEY`、`AGENTSMITH_LLM_BASE_URL`、`AGENTSMITH_LLM_MODEL`、`AGENTSMITH_LLM_PROVIDER`。

## 八、完整执行流程示例

用户输入：「帮我修复登录页的表单验证 bug」

```
 1. [task_router] 关键词"修复""bug" → Bug Fix 路由
 2. [skill_chain] 加载 Bug Fix 技能链：
    sde-debug → planning → testing-strategy → change-validation → code-review
 3. [sde-debug] ReAct 循环 → 根因分析 → SkillRubricGate + RootCauseGate 通过
 4. [planning] ReAct 循环 → 修复计划 → SkillRubricGate + PlanningGate 通过
 5. [testing-strategy] ReAct 循环 → 测试策略 → SkillRubricGate + TestGate 通过
 6. [change-validation] ReAct 循环 → 执行修复 + 运行测试
    → ValidationGate 失败 → backtrack 到 planning
    → Failure Loop Guard: 第 1 次，常规回退
    → 重新 planning → 重新 validation → 通过
 7. [code-review] ReAct 循环 → 自审 → SkillRubricGate + ReviewGate 通过
 8. UserPreferenceLearner.observe() 产出稳定信号
 9. memory lifecycle hook → save_conversation_memory() 写证据并按需编译
10. Dream 检查是否触发审核式记忆整理
11. 返回结果
```

## 九、单 Smith 与声明式领域身份

Smith 始终是用户设备上唯一运行中的 Agent。领域能力不通过创建新的档案、进程或
数据库记录实现，而是由 `agents/identities/*.yaml` 声明：

| 概念 | 当前定义 |
|---|---|
| Smith Profile | 唯一运行时档案：`~/.agent-smith/agent/` |
| Identity | 一次任务的领域 prompt、工具/技能 allowlist 与 intent→pipeline 路由 |
| Pipeline | `agents/pipelines/*.yaml` 中定义的节点、门禁与回退边 |
| Session | 对话线程；首条消息选中或显式指定 identity 后固定 |
| Memory / checkpoint | `identity-state/<identity-id>/`，防止法务、财务等领域状态互相污染 |

启动时会加载并校验身份目录。新增领域的最小闭环是：新增一个 identity YAML、它引用
的 pipeline YAML，以及需要时的 SKILL.md。`task_router.py` 不含领域词表，也无需为新
身份或新 pipeline 改动。

### 9.1 反模式清单

- **不恢复多 Agent 或按角色创建运行档案的模型**。
- **不把领域身份实现为新的数据库 profile 或独立进程**。
- **不在 Python 中硬编码法务、财务等领域 intent**；它们属于 YAML 内容。
- **不让 engine 理解 HTTP、数据库或产品层 profile 管理**；它只接收运行时上下文和已加载的 catalog。
