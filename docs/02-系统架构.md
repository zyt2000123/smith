# 02 · 系统架构

## 核心设计哲学

**宏观确定性控制，微观 LLM 自由度。**

代码（规则引擎、DAG、门禁）管流程，LLM 只在 ReAct 循环里获得「做事」的自由。这避免了纯 LLM Agent 的不可控问题。

运行定位上，Agent-Smith 是 **terminal-native / shell-native** 的本地进程 Agent。`shell` 是一等执行能力；`read_file`、`write_file`、`grep`、`git_ops`、`web_fetch` 等结构化工具，是对 shell/file/git/web 高频操作的安全封装。安全边界按能力划分（文件、网络、Git 副作用、删除、写入、secret 访问），而不是按工具名粗暴区分。

> 过渡说明：产品定位已经收口为“单常驻个人助手 Agent”。正式目标模型见 `docs/13-单常驻Agent架构重梳理.md`：Smith 的运行时档案应是 `~/.agent-smith/agent/`。当前 server / app / SQLite / 运行时目录里仍保留 `employee` / `employees` 命名，这是 legacy compatibility 壳，不是新的架构心智。

## 一、核心分层架构

```
shell/ ── HTTP/SSE ──→ server/ ── import ──→ engine/ ── import ──→ common/
                                │                      ↑
                                └──── 读取 ────→ agents/ ┘
```

依赖方向单向不可逆。`common` 在最底层不依赖任何其他层。`agents/` 是纯内容，被 engine 加载。当前现役前端是 `shell/`；历史 macOS 原生端 `app/` 已暂缓并从当前工作树移除。

| 层 | 目录 | 职责 | 语言 | 关键约束 |
|---|---|---|---|---|
| 基础设施 | `common/` | 配置、数据库、文件系统、日志 | Python | 零业务逻辑，不依赖任何其他层 |
| 执行框架 | `engine/` | 自研 Agent 框架（LLM 调用、执行引擎、记忆、安全） | Python | 不知道 FastAPI、不知道「Agent 实例管理」概念 |
| 内容定义 | `agents/` | Smith 模板、内置技能、内置工具、安全规则 | .md / .yaml / .json + Python | 能力扩展靠 skill/tool/knowledge，不靠新建 Agent |
| 平台后端 | `server/` | 本机后端（Agent Profile、会话、任务、API） | Python | 薄壳路由 → 服务编排 → Repository 持久化 |
| 终端前端 | `shell/` | Ink/React 终端工作台 | TypeScript | 通过 HTTP/SSE 与 server 通信，不直接调用 engine |

**分层测试**：把 FastAPI 换成 Flask，只需重写 `server/`——engine、agents、common 完全不动。把终端前端换成 Web/macOS 客户端，也只改前端层。

## 二、各层详细职责

### 2.1 common/ — 公共基础设施

心智定位：`common/` 是最底层基础设施，不承载产品流程。它可以知道“文件在哪、配置怎么合并、SQLite 怎么连接”，不应该知道“业务怎么跑、API 怎么编排、Smith 生命周期怎么推进”。

`database.py` 只负责 SQLite 连接、WAL 和外键设置；业务 schema、Repository 和 profile 文件生命周期属于 `server/app/infrastructure/`。

```
common/
├── config.py            # 路径常量
├── config_loader.py     # LLM 配置合并（环境变量 → 平台级 → Smith 出厂级 → profile 级 → 会话级）
├── database.py          # SQLite 连接
├── paths.py             # AppPaths 路径对象（目标路径 + legacy path）
├── yaml_utils.py        # YAML / JSON 读写
└── pyproject.toml       # 依赖: pyyaml, aiosqlite
```

**数据库 schema**（`server/app/infrastructure/schema.py`）：

| 表 | 职责 |
|---|---|

#### 2.2.1 四层执行引擎

用户发一条消息后经历四层处理：

```
用户消息："帮我修复登录页的表单验证 bug"
        │
   ┌────▼──────────────────────────────┐
   │ 第 1 层：任务路由（task_router）     │  代码控制
   │ "修复" + "bug" → Bug Fix 路由      │  规则引擎，不用 LLM
   └────┬──────────────────────────────┘
        │
   ┌────▼──────────────────────────────┐
   │ 第 2 层：技能链 DAG（skill_chain）  │  代码控制
   │ sde-debug → planning → testing    │  有向图，强制顺序
   │ → change-validation → code-review │
   └────┬──────────────────────────────┘
        │ 对每个技能节点
   ┌────▼──────────────────────────────┐
   │ 第 3 层：ReAct 循环（agent_loop）   │  LLM 控制
   │ 思考 → 调工具 → 观察 → 思考 → ... │  这里 LLM 有自由度
   └────┬──────────────────────────────┘
        │ 每次调工具前
   ┌────▼──────────────────────────────┐
   │ 第 4 层：安全护栏（tool_guard）     │  代码控制
   │ 正则匹配 24 条规则，拦截危险命令    │  7 类安全规则
   └───────────────────────────────────┘
```

第 1、2、4 层不用 LLM——「先规划再编码」「修 bug 必须先调试」「不能 rm -rf」是确定性规则。LLM 只在第 3 层获得自由。

**三条路由**（`task_router.py`）：

| 输入特征 | 路由结果 | 处理方式 |
|---|---|---|
| 关键词 fix/bug/error + stack trace | `TaskType.BUGFIX` | 走 Bug Fix 技能链 |
| 关键词 feature/add/implement + 需求描述 | `TaskType.FEATURE` | 走 Feature 技能链 |
| 简单问答 / 代码解释 | `TaskType.DIRECT` | 跳过技能链，直接 ReAct |

**技能链定义**（`skill_chain.py`）：

```
Feature 路由：
  planning → architecture(条件:3+文件/跨模块) → testing-strategy → change-validation → code-review

Bug Fix 路由：
  sde-debug → planning → testing-strategy → change-validation → code-review

Direct 路由：
  无技能链，直接进入 agent_loop 的 ReAct 循环
```

硬规则：Bug Fix 必须使用 sde-debug 和 testing-strategy，不可替代；architecture 仅在变更涉及 3+ 文件或跨模块时触发；code-review 始终是最后一步，不可跳过。

#### 2.2.2 门禁系统（`gate.py`）

双层门禁：每个技能执行后先过 Skill Rubric Gate（通用质量检查），再过节点专属 Gate。

| 类型 | 门禁 | 通过条件 |
|---|---|---|
| 流程门禁 | PlanningGate | 步骤 >= 3 + 有验证点 |
| | RootCauseGate | 有根因声明 + 证据支撑 |
| | DesignGate | 涉及文件 + 数据流 + 依赖关系 |
| | TestGate | 测试用例 >= 2 + 边界覆盖 |
| | ValidationGate | 有执行证据 + pass/fail 结果 |
| | ReviewGate | 有 P0/P1 分级 + 可操作发现 |
| 通用门禁 | SkillRubricGate | 长度 > 50 + 代码块/文件引用 + 无错误标记 |
| 交付门禁 | GitWorktreeGate | Worktree 就绪 + 无冲突 |
| | PRGate | Conventional Commit 格式 + 无敏感文件 |
| | TestDeliveryGate | 测试实际运行 + 有 pass/fail 计数 |

#### 2.2.3 回退 + Failure Loop Guard（`backtrack.py`）

```
回退路径：
  Implementation 失败 → 回退到 Planning
  Design 失败       → 回退到 Understand
  Verify 失败       → 回退到 Planning

Failure Loop Guard：
  1. 记录失败签名 = hash(error_type + context_hash)
  2. 同一签名出现 2 次 → 换策略（不是简单重试）
  3. 最多 2 种替代策略 → 全部失败 → blocked，请求人工介入
```

不是限制重试次数，而是识别「用同样方式犯同样错误」。

#### 2.2.4 System Prompt 9 层组装（`assembler.py`）

| 层 | 来源 | 职责 |
|---|---|---|
| 1 | role.md | 核心使命（最高优先级） |
| 2 | style.md | 行为风格 |
| 3 | workflow.md | 工作流 SOP + 门禁定义 |
| 4 | toolbox.md + 工具注册表 | 可用工具声明 + 使用规范 |
| 5 | 技能注册表 | 可用技能摘要 + 使用策略 |
| 6 | context.md | 用户偏好（交互中自动学习填充） |
| 7 | output_style.md | 输出格式规范 |
| 8 | memory/ | 记忆摘要（project 优先于 agent） |
| 9 | 运行时上下文 | Agent Profile / session_id / 环境 |

层 1-3 定义 Smith 的稳定身份和全局工作规约，层 4-5 提供工具与 skill 目录，层 6-9 提供用户偏好、输出规范、长期记忆和瞬时运行时信息。具体场景 SOP 不再放在模板 JSON 中，而是下沉到 `agents/skills/`。

#### 2.2.5 记忆系统

目标目录：

```
~/.agent-smith/agent/memory/
├── agent/          # 当前实现：个人/Agent 级记忆（目标语义为 personal memory）
├── project/        # 项目级记忆（项目特定，搜索时优先）
├── conversations/  # 对话记忆
└── recent.jsonl    # 最近对话索引
```

当前兼容实现仍可能落在：

```
~/.agent-smith/employees/<id>/memory/
```

后者只表示迁移前 profile path，不再表示“多个数字员工”的产品模型。

**Dream 机制**（`dream.py`）：每 5 次对话自动触发 DreamConsolidator：
1. 过滤秘密（8 种模式）
2. 剪枝（30 天无访问）
3. 去重（70% 重叠）
4. 模式提取（3+ 共主题）

**偏好学习**（`user_learner.py`）：4 维启发式检测（语言、简洁度、技术水平、代码风格）。同一特征观察 3 次以上才写入 context.md，仅填充 `{{to_be_learned}}` 占位符，不覆盖用户手动编辑。

#### 2.2.6 技能自进化

| 类型 | 位置 | 权限 |
|---|---|---|
| 内置技能 | `agents/skills/` | 只读 |
| Smith 自装技能（目标） | `~/.agent-smith/agent/skills/` | 可进化 |
| Agent 技能（legacy） | `~/.agent-smith/employees/<id>/skills/` | 迁移期兼容路径 |

SkillStore 版本管理：保留最近 10 个版本，支持 rollback 和 diff。skill_manage 工具 7 个操作：list / get / create / edit / patch / versions / rollback。内置技能只读保护。

#### 2.2.7 安全体系

四层安全：

1. **工具调用层** — 24 条正则规则，7 类拦截（command_injection, resource_abuse, code_execution, network_abuse, sensitive_file_access, privilege_escalation, shell_evasion）
2. **文件访问层** — 路径白名单 + 敏感目录黑名单，覆盖 shell、读写、搜索、枚举、Git worktree 等路径型能力
3. **记忆安全层** — 8 种秘密过滤
4. **交付安全层** — PRGate 禁止敏感文件

### 2.3 agents/ — 纯内容层

`agents/` 是内容资产库，不是运行实例集合。当前内置身份种子是 `agents/smith/`，新增能力优先放到 `agents/skills/`、`agents/tools/` 或知识注入资料里，而不是新增 Agent 角色。

### 2.5 shell/ — Ink 终端前端

当前现役前端是 `shell/`，负责用户交互、会话选择、SSE 渲染、本地 server 启动和 CLI 命令入口。它只通过 HTTP/SSE 调用 `server/`，不直接 import `engine/`。

```
shell/src/
├── index.tsx          # Ink UI 入口
├── api.ts             # HTTP/SSE API 客户端
├── bridge.ts          # 后端启动与协议桥接

├── hud.tsx            # 状态栏/HUD
├── store.ts           # 前端状态
```

macOS 原生端不再作为当前主线；如未来恢复，应作为新的前端客户端重新接入 `server/`，不改变 engine/agents/common 的依赖方向。

## 三、LLM 配置四层覆盖

| 层级 | 位置 | 优先级 | 场景 |
|---|---|---|---|
| 环境变量/平台级 | 环境变量 `AGENTSMITH_LLM_*` + `~/.agent-smith/config.yaml` | 最低 | 全局默认 |
| Smith 出厂级 | `agents/smith/config.yaml` | ↑ | Smith 出厂默认配置 |
| Smith 级（目标） | `~/.agent-smith/agent/config.yaml` | ↑ | Smith 专属配置 |
| Agent 级（legacy） | `~/.agent-smith/employees/<id>/config.yaml` | ↑ | 迁移期兼容配置 |
| 会话级 | API 参数传入 | 最高 | 临时切换模型 |

解析逻辑（`config_loader.py`）：从上到下逐层 deep merge，下层**覆盖**上层已有字段，未填字段**继承**上层。

支持的环境变量：

| 环境变量 | 配置字段 |
|---|---|
| `AGENTSMITH_LLM_API_KEY` | `api_key` |
| `AGENTSMITH_LLM_BASE_URL` | `base_url` |
| `AGENTSMITH_LLM_MODEL` | `model` |
| `AGENTSMITH_LLM_PROVIDER` | `provider` |

配置示例：

```yaml
# ~/.agent-smith/config.yaml（平台级）
llm:
  provider: openai
  api_key: sk-xxx
  base_url: https://newapi.elevatesphere.com/v1
  model: glm-4.7

# ~/.agent-smith/agent/config.yaml（Smith 级目标路径）
llm:
  model: gpt-4o          # 只覆盖模型，key/url 继承平台级
tools:
  enabled: [read_file, write_file, shell]
  disabled: [web_fetch]  # 这个 Agent 不需要抓网页
```

## 八、完整执行流程示例

用户给前端工程师发消息：「帮我修复登录页的表单验证 bug」

```
 1. [task_router] 关键词"修复""bug" → Bug Fix 路由
 2. [skill_chain] 加载 Bug Fix 技能链：
    sde-debug → planning → testing-strategy → change-validation → code-review
 3. [sde-debug] ReAct 循环 → 根因分析 → SkillRubricGate + RootCauseGate 通过
 4. [planning] ReAct 循环 → 修复计划 → SkillRubricGate + PlanningGate 通过
 5. [testing-strategy] ReAct 循环 → 测试策略 → SkillRubricGate + TestGate 通过
 6. [change-validation] ReAct 循环 → 执行修复 + 运行测试
    → ValidationGate 失败 → backtrack 到 planning
    → Failure Loop Guard: 第 1 次，常规回退
    → 重新 planning → 重新 validation → 通过
 7. [code-review] ReAct 循环 → 自审 → SkillRubricGate + ReviewGate 通过
 8. save_conversation_memory()
 9. UserPreferenceLearner.observe()
10. 每 5 次 → Dream 整理记忆
11. 返回结果
```
