# 11 · 整体架构与设计思路

> 2026-07-04。Agent-Smith 完整系统设计文档。

## 核心设计哲学

**宏观确定性控制，微观 LLM 自由度。**

代码（规则引擎、DAG、门禁）管流程，LLM 只在 ReAct 循环里获得"做事"的自由。这避免了纯 LLM agent 的不可控问题。

---

## 一、五层分离架构

```
macos-app/ ──HTTP/SSE──→ server/ ──import──→ engine/ ──import──→ common/
                            │                    ↑
                            └── 读取 ──→ agents/  ┘
```

依赖方向不可逆。

| 层 | 职责 | 关键约束 |
|---|---|---|
| `common/` | 基础设施（配置、数据库、文件系统、日志） | 零业务逻辑，不依赖任何其他层 |
| `engine/` | Agent 执行框架（LLM 调用、执行引擎、记忆、安全） | 不知道 FastAPI、不知道"员工"概念 |
| `agents/` | Agent 内容定义（角色模板、技能、工具、安全规则） | 纯文件（.md/.yaml/.json），新加 agent 只碰这里 |
| `server/` | 平台后端（员工管理、会话、任务、API） | 薄壳路由 → 服务编排 → Repository 持久化 |
| `macos-app/` | 原生前端 | Swift/SwiftUI，通过 HTTP/SSE 与 server 通信 |

**分层测试**：如果把 FastAPI 换成 Flask，只需重写 server/——engine、agents、common 完全不动。把 macOS 前端换成 Web 前端，也只改前端层。

## 二、执行引擎——四层嵌套

用户发一条消息后经历四层处理：

```
用户消息："帮我修复登录页的表单验证 bug"
        │
   ┌────▼─────────────────────────────┐
   │ 第 1 层：任务路由（task_router）    │  代码控制
   │ "修复" + "bug" → Bug Fix 路由     │  规则引擎，不用 LLM
   └────┬─────────────────────────────┘
        │
   ┌────▼─────────────────────────────┐
   │ 第 2 层：技能链 DAG（skill_chain） │  代码控制
   │ sde-debug → planning → testing   │  有向图，强制顺序
   │ → change-validation → code-review│
   └────┬─────────────────────────────┘
        │ 对每个技能节点
   ┌────▼─────────────────────────────┐
   │ 第 3 层：ReAct 循环（agent_loop）  │  LLM 控制
   │ 思考 → 调工具 → 观察 → 思考 → ...│  这里 LLM 有自由度
   └────┬─────────────────────────────┘
        │ 每次调工具前
   ┌────▼─────────────────────────────┐
   │ 第 4 层：安全护栏（tool_guard）    │  代码控制
   │ 正则匹配 24 条规则，拦截危险命令   │  7 类安全规则
   └──────────────────────────────────┘
```

第 1、2、4 层不用 LLM，因为"先规划再编码"、"修 bug 必须先调试"、"不能 rm -rf"是确定性规则。LLM 只在第 3 层获得自由。

### 2.1 三条路由

| 输入 | 路由结果 | 处理方式 |
|---|---|---|
| "帮我修复 bug" | `TaskType.BUGFIX` | 走 Bug Fix 技能链 |
| "帮我加个功能" | `TaskType.FEATURE` | 走 Feature 技能链 |
| "解释一下这段代码" | `TaskType.DIRECT` | 跳过技能链，直接 ReAct |

对应文件：`engine/execution/task_router.py`

### 2.2 技能链定义

**Feature 路由**：
```
planning → architecture(条件:3+文件/跨模块) → testing-strategy → change-validation → code-review
```

**Bug Fix 路由**：
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix 必须使用 sde-debug 和 testing-strategy，不可替代
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review 始终是最后一步，不可跳过

对应文件：`engine/execution/skill_chain.py`

### 2.3 ReAct 循环

```python
async def _react_loop(llm, messages, tool_registry, tool_guard, max_iters=20):
    for _ in range(max_iters):
        response = await llm.chat(messages, tools=tools)
        if not response.has_tool_calls:
            return response.text
        for tc in response.tool_calls:
            guard_result = tool_guard.check(tc)
            if not guard_result.allowed:
                append("[BLOCKED]")
                continue
            result = await tool_registry.execute(tc)
            append(result)
    return "Max iterations reached."
```

提供同步和流式两个版本。对应文件：`engine/execution/agent_loop.py`

## 三、门禁系统

### 3.1 双层门禁

```
技能执行 → Skill Rubric Gate（通用质量检查）
              │
              ├── 通过 → 节点专属 Gate
              │              │
              │              ├── 通过 → 下一个技能节点
              │              └── 失败 → FailureLoopGuard 判断
              │
              └── 失败 → 重试（最多 3 次）
                    ├── 第 1 次：带 retry_hint 重执行
                    ├── 第 2 次：换策略执行
                    └── 第 3 次：blocked
```

### 3.2 门禁清单（12 个）

| 类型 | 门禁 | 检查内容 |
|---|---|---|
| 流程门禁 | PlanningGate | 步骤 ≥3 + 有验证点 |
| | TestGate | 测试用例 ≥2 + 边界覆盖 |
| | ValidationGate | 有执行证据 + pass/fail 结果 |
| | ReviewGate | 有 P0/P1 分级 + 可操作发现 |
| | RootCauseGate | 有根因声明 + 证据支撑 |
| | DesignGate | 涉及文件 + 数据流 + 依赖关系 |
| 通用门禁 | SkillRubricGate | 长度 >50 + 代码块/文件引用 + 无错误标记 |
| 交付门禁 | GitWorktreeGate | Worktree 就绪 + 无冲突 |
| | PRGate | Conventional Commit 格式 + 无敏感文件 |
| | TestDeliveryGate | 测试实际运行 + 有 pass/fail 计数 |

对应文件：`engine/execution/gate.py`

### 3.3 回退 + Failure Loop Guard

```
回退路径：
  Implementation 失败 → 回退到 Planning
  Design 失败       → 回退到 Understand
  Verify 失败       → 回退到 Planning

Failure Loop Guard：
  1. 记录失败签名 = hash(error_type + context_hash)
  2. 同一签名出现 2 次 → 换策略（不是简单重试）
  3. 最多 2 种替代策略 → 全部失败 → blocked
```

不是限制重试次数，而是识别"用同样方式犯同样错误"。

对应文件：`engine/execution/backtrack.py`

## 四、System Prompt 12 层组装

| 层 | 来源 | 职责 |
|---|---|---|
| 1 | role.md | 核心使命 |
| 2 | style.md | 行为风格 |
| 3 | workflow.md | 工作流 SOP |
| 4 | toolbox.md + 工具注册表 | 可用工具 |
| 5 | 技能注册表 | 可用技能 |
| 6 | expertise.json | 核心能力声明 |
| 7 | traits.json | 工作风格标签 |
| 8 | pipeline.json | 交付承诺 |
| 9 | context.md | 用户偏好（自动学习填充） |
| 10 | output_style.md | 输出格式规范 |
| 11 | memory/ | 记忆（project 优先于 agent） |
| 12 | 运行时上下文 | employee_id / session_id / 环境 |

层 1-3 定义"你是谁"，层 4-5 定义"你能做什么"，层 6-8 是结构化元数据，层 9-11 随时间变化，层 12 是瞬时运行时信息。

对应文件：`engine/prompt/assembler.py`

## 五、记忆系统

### 5.1 存储架构

```
~/.agent-smith/employees/<id>/memory/
├── agent/          # Agent 级记忆（跨项目通用）
├── project/        # 项目级记忆（项目特定，搜索优先）
├── conversations/  # 对话记忆
└── recent.jsonl    # 最近对话索引
```

### 5.2 Dream 机制

```
对话发生 → save_conversation_memory() → 每 5 次触发 Dream
                                              │
                                    ┌─────────▼──────────┐
                                    │   DreamConsolidator  │
                                    │  1. 过滤秘密（8 种）  │
                                    │  2. 剪枝（30天无访问）│
                                    │  3. 去重（70%重叠）   │
                                    │  4. 模式提取（3+共主题）│
                                    └──────────────────────┘
```

对应文件：`engine/memory/dream.py`、`engine/memory/store.py`

## 六、技能自进化

| 类型 | 位置 | 权限 |
|---|---|---|
| 内置技能 | `agents/skills/` | 只读 |
| 员工技能 | `~/.agent-smith/employees/<id>/skills/` | 可进化 |

SkillStore 版本管理：保留最近 10 个版本，支持 rollback 和 diff。

skill_manage 工具 7 个操作：list / get / create / edit / patch / versions / rollback。内置技能只读保护。

对应文件：`engine/skill/store.py`、`agents/tools/skill_manage.py`

## 七、用户偏好自动学习

4 维启发式检测：语言、简洁度、技术水平、代码风格。

置信计数器：同一特征观察 3 次以上才写入 context.md。仅填充 `{{to_be_learned}}` 占位符，不覆盖用户手动编辑。

对应文件：`engine/memory/user_learner.py`

四层安全：
1. **工具调用层** — 24 条正则规则，7 类拦截

## 十一、员工生命周期

```
agents/templates/<role>/  →  创建时复制  →  ~/.agent-smith/employees/<id>/
（只读模板）                              （运行时读写 + 记忆积累 + 技能进化）
```

新加 Agent 只需在 `agents/templates/` 下创建目录和文件，不改任何代码。

现有 8 个角色模板：backend-engineer、frontend-engineer、devops-engineer、test-engineer、data-analyst、product-manager、content-ops、ui-designer。

## 十二、LLM 配置

四层覆盖，优先级从低到高：平台级 → 模板级 → 员工级 → 会话级。

下层覆盖上层已有字段，未填字段继承上层。

对应文件：`common/config_loader.py`

## 十三、完整执行流程示例

用户给前端工程师发消息："帮我修复登录页的表单验证 bug"

```
 1. [task_router] "修复" + "bug" → Bug Fix 路由
 2. [skill_chain] sde-debug → planning → testing → validation → review
 3. [sde-debug] ReAct → SkillRubricGate ✓ → RootCauseGate ✓
 4. [planning] ReAct → SkillRubricGate ✓ → PlanningGate ✓
 5. [testing] ReAct → SkillRubricGate ✓ → TestGate ✓
 6. [validation] ReAct → ValidationGate ✗ → backtrack → re-planning → ✓
 7. [code-review] ReAct → SkillRubricGate ✓ → ReviewGate ✓
 8. save_conversation_memory()
 9. UserPreferenceLearner.observe()
10. 每 5 次 → Dream 整理记忆
11. 返回结果
```
