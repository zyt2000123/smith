# 09 · 架构设计规范

> 2026-07-04 定稿。基于 QoderWake 逆向分析 + octok-platform DDD 分层，确定 Agent-Smith 的顶层架构、分层职责、依赖方向和核心设计。

## 一、顶层结构

```
Agent-Smith/
├── common/          # ⚙️  公共基础设施（零业务逻辑）
├── engine/          # 🔴  Agent 引擎（自研框架，独立可复用）
├── agents/          # 🟡  Agent 定义（纯内容：模板 + 技能 + 工具 + 规则）
├── server/          # 🔵  平台后端（FastAPI，管理 + API）
├── macos-app/       # 🟢  前端（SwiftUI 原生 macOS）
└── docs/            # 📖  文档
```

| 目录 | 性质 | 语言 | 新加 Agent 碰否 |
|---|---|---|---|
| common | 公共基础设施 | Python | 不碰 |
| engine | 自研 Agent 框架 | Python | 不碰 |
| agents | Agent 内容定义 | .md / .yaml / .json + Python provider | **只碰这里** |
| server | 平台后端 | Python (FastAPI) | 不碰 |
| macos-app | 原生前端 | Swift (SwiftUI) | 不碰 |

## 二、依赖方向

```
macos-app/  ── HTTP/SSE ──→  server/  ── import ──→  engine/  ── import ──→  common/
                               │                         ↑
                               └──── 读取 ────→  agents/  ┘
                                                (被 engine 加载)
```

单向依赖，不可逆。`common` 在最底层不依赖任何模块。`agents/` 是纯内容，不依赖代码。

```yaml
1. [步骤 1] → 验证: [检查点]

```

## 七、LLM 配置 — 四层覆盖

| 层级 | 位置 | 优先级 | 场景 |
|---|---|---|---|
| 平台级 | `~/.agent-smith/config.yaml` | 最低 | SwiftUI 设置页 / API 配置，全局默认 |
| 模板级 | `agents/templates/xxx/config.yaml` | ↑ | 模板建议的默认模型 |
| 员工级 | `~/.agent-smith/employees/<id>/config.yaml` | ↑ | 该员工专属配置 |
| 会话级 | API 参数传入 | 最高 | 临时切换模型 |

解析逻辑：从上到下逐层合并，下层**覆盖**上层已有字段，不填的字段**继承**上层。

```yaml
# ~/.agent-smith/config.yaml（平台级）
llm:
  provider: openai
  api_key: sk-xxx
  base_url: https://newapi.elevatesphere.com/v1
  model: glm-4.7

# ~/.agent-smith/employees/<id>/config.yaml（员工级）
llm:
  model: gpt-4o          # 只覆盖模型，key/url 继承平台级

tools:
  enabled: [read_file, write_file, shell]
  disabled: [web_fetch]  # 这个员工不需要抓网页
```

## 八、执行引擎 — 混合 DAG + ReAct

执行引擎采用两层混合架构，借鉴 QoderWake 的 Harness-First 设计：

| 层 | 模式 | 控制什么 |
|---|---|---|
| 宏观编排 | DAG | 技能链顺序、门禁、回退路径 — **代码控制，确定性** |
| 微观执行 | ReAct | 单个技能/步骤内怎么思考和用工具 — **LLM 控制，灵活性** |

### 8.1 任务路由

```
收到任务
  ├── Bug Fix   → 走 Bug Fix 路由
  ├── Feature   → 走 Feature 路由
  └── Direct    → 直接回复（跳过技能链）

判断依据：
  ├── 关键词（fix / bug / error / feature / add / implement）
  ├── 上下文线索（stack trace / 新需求描述）
  └── 用户显式指定
```

### 8.2 强制技能链（DAG）

```
Feature 路由：
  planning → architecture(条件:3+文件/跨模块) → testing-strategy → change-validation → code-review

Bug Fix 路由：
  sde-debug → planning → testing-strategy → change-validation → code-review
```

每个节点内部运行 ReAct 循环。节点之间由门禁控制流转。

### 8.3 门禁系统

| 门禁 | 位置 | 通过条件 | 失败处理 |
|---|---|---|---|
| Understanding Gate | 理解阶段末 | 能准确复述需求 + 识别边界 | 回退重新理解 |
| Root Cause Gate | 调试阶段末 | 有证据支持的根因 | 回退到 Reproduce |
| Planning Gate | 规划阶段末 | 步骤完整 + 有验证点 | 重新规划 |
| Contract Alignment Gate | 实现前 | 方案与契约一致 | 回退到 Planning |
| Validation Gate | 验证阶段末 | 测试通过 + 无回归 | 回退到 Planning |
| Review Gate | Code Review 末 | 无 P0/P1 问题 | 修复后重新 Review |
| Skill Rubric Gate | 每个技能执行后 | 输出满足质量标准 | 最多重试 2 次 |

### 8.4 回退 + Failure Loop Guard

```
回退路径：
  Implementation 失败 → 回退到 Planning
  Design 失败       → 回退到 Understand
  Verify 失败       → 回退到 Planning

Failure Loop Guard：
  1. 跟踪失败签名（错误类型 + 上下文 hash）
  2. 同一签名出现 2 次 → 停止当前策略
  3. 尝试最多 2 种替代策略
  4. 替代也失败 → 声明 blocked，请求人工介入
```

## 九、System Prompt 组装顺序

8 层有序组装，借鉴 QoderWake 的 13 层精简而来：

| 层 | 来源 | 职责 |
|---|---|---|
| 1 | role.md | 核心使命（最高优先级） |
| 2 | style.md | 行为风格 |
| 3 | workflow.md | 工作流 SOP + 门禁定义 |
| 4 | toolbox.md + 工具注册表 | 可用工具声明 + 使用规范 |
| 5 | 技能注册表 | 可用技能摘要 + 技能使用策略 |
| 6 | context.md | 用户偏好（交互中自动学习） |
| 7 | memory/ 索引 | 记忆摘要 |
| 8 | 运行时上下文 | 员工 ID / 会话 ID / 环境变量 |

## 十、server/ — 平台后端

```
server/
├── pyproject.toml              # 依赖 common + engine
└── app/
    ├── main.py                 # FastAPI 入口
    ├── domain/                 # 平台级领域模型
    │   ├── employee.py         #   Employee（agent 实例化后的管理对象）
    │   ├── session.py          #   Session, Message
    │   ├── task.py             #   Task, TaskType, TaskStatus
    │   └── connector.py        #   Connector（Hub MCP 等）
    ├── services/               # 业务编排
    │   ├── employee_service.py
    │   ├── session_service.py
    │   ├── task_service.py
    │   ├── template_service.py
    │   └── config_service.py
    ├── infrastructure/         # 持久化 + 外部连接
    │   ├── employee_fs.py
    │   ├── repositories/
    │   │   ├── employee_repo.py
    │   │   ├── session_repo.py
    │   │   └── task_repo.py
    │   └── hub_connector.py
    └── routers/                # HTTP API（薄壳，只做参数提取 + 调 service）
        ├── employees.py
        ├── sessions.py
        ├── tasks.py
        ├── templates.py
        ├── files.py
        └── config.py
```

Router 层规范（借鉴 octok-platform 的 Controller 瘦身规范）：

- Router 只做：提取参数 → 调 service → 返回结果
- Router 不做：业务判断、数据查找、条件分支、错误抛出
- 业务逻辑全在 service 层
