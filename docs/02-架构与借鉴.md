# 02 · 架构与借鉴

> 2026-07-03 修订：产品形态已从「多 Agent 圆桌工作台」收敛为「本地Agent工作台」。QoderWake 作为产品形态和架构参考，LLM-Wiki-Knowledge-Hub 作为前端技术栈参考，AgentScope 作为运行时参考；不复用第三方产品代码或品牌资产。

## 总体架构

架构方案待 Phase 0 spike 后确定。当前有两个候选方案：

### 方案 A：daemon + 浏览器（QoderWake 方案）

```
本地 daemon 进程
    │ 启动时监听 localhost:<port>
    ▼
Python 本地 server（FastAPI + AgentScope）
    ├─ Agent运行时
    ├─ Agent存储：~/.agent-smith/employees/<id>/
    ├─ 本地任务：对话任务 / 自动任务 / 工作记录
    ├─ 本地能力：文件、命令、浏览器等工具白名单
    ├─ 权限：工作目录限制 + 工具级询问/允许/拒绝
    └─ 连接器：Hub MCP / Hub 检索 API / 后续其他 MCP
    ▲
    │ HTTP + SSE
浏览器访问 http://localhost:<port>
    └─ React 前端（Vite + Tailwind + shadcn/Base UI）
```

优势：不打包 Chromium（省 ~150MB），分发轻量（shell 脚本安装），前端就是标准 Web 应用。
劣势：没有原生窗口管理，依赖用户浏览器，托盘/通知/系统集成需额外方案。

### 方案 B：Electron 壳（原方案）

```
Electron 桌面壳
    │ spawn + 本地 server 握手
    ▼
Python 本地 server（FastAPI + AgentScope）
    ├─ 同上
    ▲
    │ HTTP + SSE
Electron 内嵌 React 前端
```

优势：原生窗口、托盘、系统通知、自动更新，桌面体验完整。
劣势：打包体积大，签名/公证流程复杂，多平台维护成本高。

**Phase 0 需要 spike 两种方案的启动体验和分发复杂度后再定。**

前端负责Agent产品形态；本地 server 负责Agent运行、任务执行、权限和持久化；Hub 只作为团队知识工具来源。

## 技术栈

| 层 | 技术 | 说明 |
|---|---|---|
| 桌面壳 | 待定（daemon + 浏览器 或 Electron） | Phase 0 spike 后确定，Mac 优先 |
| 前端 | React 19 + Vite + Tailwind v4 + shadcn/Base UI + lucide | 复用 LLM-Wiki-Knowledge-Hub 的前端技术栈和组件习惯 |
| 本地 server | Python + FastAPI + SSE | 与前端本地 HTTP 通信 |
| Agent 运行时 | AgentScope | 先跑单Agent ReActAgent，后续再接 MsgHub 多Agent协作 |

| 知识连接 | Hub MCP 或 Hub 检索 API | Phase 0 spike 后定默认路径 |

## 借鉴对象和边界

### QoderWake：产品形态与架构参考

> QoderWake 是阿里巴巴旗下 Qoder 团队（国内品牌：通义灵码）于 2026 年 5 月推出的 AI Agent平台。以下基于 2026-07-03 的深度技术调研。

#### QoderWake 实际技术栈（网络调研 + 本地逆向工程，2026-07-03）

| 维度 | QoderWake 方案 |
|---|---|
| 桌面壳 | **Swift/AppKit 菜单栏壳**（647KB）+ **定制 Bun v1.3.14 守护进程**（110MB），非 Electron/Tauri |
| 核心运行时 | TypeScript/JavaScript 跑在定制 Bun 分支上（Zig 编译 + JavaScriptCore 引擎） |
| Agent 框架 | 自研 Harness-First Architecture（思考与执行分离、双层验证、Critic-Refiner 机制），非 AgentScope/LangChain |
| 前端 | **React + Vite + Tailwind CSS v4.1.18 + Ant Design (antd)**，SWR 数据获取，Shiki/Marked/Mermaid/KaTeX |
| 通信协议 | HTTP + SSE + ACP（JSON-RPC 2.0 over stdio），Chrome 扩展通过 nativeMessaging |
| MCP 支持 | 4 种传输：STDIO / SSE / Streamable HTTP / SDK；守护进程自身暴露 MCP 端点 |
| 存储 | **SQLite 76 表**（WAL 模式，32 次 schema 迁移）+ **sqlite-vec 向量搜索** + **Git 版本化** Memory/Skills |
| 底层模型 | Qwen3.7 为主，兼容 GLM-5.1、DeepSeek-V4-Pro、Kimi-K2.6 等国产模型（不支持 OpenAI/Anthropic） |
| 工作模式 | Ask（问答）/ Agent（自主执行）/ Quest（任务探索）+ 自定义 Skills |
| 技能打包 | SKILL.md（YAML frontmatter + Markdown 指令）、Plugin（plugin.json + skills/）、Connector（OAuth 2.0 MCP） |
| 经验系统 | 5 维：Memory / Skills / Strategy / Validation Rules / Workflows |
| 安全 | 本地虚拟环境执行、授权文件夹、删除走恢复区、红线/边界机制、dangerous_shell_commands.json 护栏 |
| 插件触发器 | Python 处理器（DingTalk / Aone / GitHub Webhooks） |
| 移动端 | Qoder Remote Control（iOS/Android），框架未确定 |
| 安装 | `curl | bash` 脚本安装至 `~/.qoderwake/` |

## 本地数据模型草案

```
~/.agent-smith/
  config.yaml
  employees/
    <employee_id>/
      config.yaml
      identity.md
      persona.md
      bible.md
      avatar.png
      memory/
        today.md
        longterm.md
      sessions/
        <session_id>.jsonl
      tasks/
        <task_id>.json
  connectors/
    hub.yaml
  sqlite/
    agent-smith.sqlite
```

说明：

- `identity.md`：Agent是谁，负责什么。
- `persona.md`：工作风格、表达方式、边界。
- `bible.md`：工作流、操作规范、验收标准。
- SQLite 保存可查询索引和状态，文件仍是Agent配置的真相源。

## 运行时能力分层

1. **Agent层**：角色、人设、记忆、默认模型、默认能力。
2. **任务层**：对话任务、自动任务、计划、状态、输出。
3. **能力层**：MCP、技能、本地工具、知识库。
4. **权限层**：工作目录、工具白名单、询问/允许/拒绝规则。
5. **审计层**：工作记录、消息记录、任务日志、来源回链。

右侧任务/能力面板对应第 2-4 层，Agent详情首页对应第 1 和第 5 层。

## 圆桌会议的位置

圆桌会议仍然有价值，但不再是 v0 的核心路径。

它应该建立在以下能力稳定之后：

- Agent创建和角色模板稳定
- 单Agent对话任务稳定
- 自动任务稳定
- 任务计划和能力面板稳定
- Agent记忆和工作记录可追踪

届时再用 AgentScope MsgHub 做多Agent协作会更自然：用户不是先进入一个「圆桌功能」，而是从已有Agent里选择多人协作。

## 风险清单

| 风险 | 应对 |
|---|---|
| 产品误入泛聊天 | 文档和原型都以Agent、任务、Agent主页为中心 |
| 能力配置过散 | MCP、技能、权限、知识库统一进入右侧任务/能力面板 |
| 本地执行风险 | v0 先做工作目录限制、工具白名单、敏感操作询问 |
| Python 打包分发不确定 | Phase 0 先做 Electron + FastAPI 打包 PoC |
| AgentScope 版本变化 | 封装 adapter，锁版本，升级走回归 |
| 过早做多Agent协作 | 圆桌放到后置，先打通单Agent完整闭环 |
