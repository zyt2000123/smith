# 01 · 产品设计与定位

## 产品定位

**Agent-Smith 是本地优先（local-first）的个人助手 Agent 工作台，初期运行在终端。**

Smith 是唯一的 Agent。它运行在本机环境中，围绕你的文件、任务、知识、技能和工作记录持续工作。通过技能系统按任务类型切换工作流，通过知识注入按需获取领域专业能力。

一句话：

> 用户在终端中与 Smith 持续协作。Smith 是唯一 Agent，技能切换工作流，知识注入补充专业能力，所有工作在本机完成，记忆跨会话积累。

## 为什么是单 Agent

多 Agent 方案（按角色拆分前端/后端/测试等子 Agent）听起来合理，但实际效果有限：

1. **LLM 的领域能力来自训练数据，不来自 role.md** — 给它加一个"你是前端工程师"的人设不会让它更会写 React
2. **真实任务不按角色分** — "改一下 API 返回格式，前端也调一下"既是前端又是后端，路由给谁？
3. **路由本身是个难题** — 分类错误比不分类更差

真正影响输出质量的是：**工具**（能不能跑命令、读文件）、**工作流**（debug 先查日志还是先看代码）、**领域知识**（React 的 hook 规则、SQL 的索引设计原则）。这些分别由工具系统、技能系统、知识注入解决，不需要多 Agent。

## Agent 模型

### Smith — 唯一 Agent

- 模板：`agents/templates/personal-assistant/`
- 角色：理解目标、整理上下文、检索信息、规划执行、交付结果
- 始终在线，用户的唯一交互对象
- 积累记忆和偏好，越用越懂你

### 三层能力扩展

```
Smith
  │
  ├── 工具系统 → 能做什么（读文件、跑命令、搜索、Git...）
  │     └── agents/tools/ 下 9 个工具 provider
  │
  ├── 技能系统 → 怎么做（工作流 SOP + 门禁）
  │     └── agents/skills/ 下 6 个 SKILL.md
  │
  └── 知识注入 → 知道什么（领域参考文档）
        └── 按需加载到 prompt context
```

- 新增"能做什么" → 加工具 provider
- 新增"怎么做" → 加 SKILL.md
- 新增"知道什么" → 加知识文档
- **不需要加新 Agent**

## 目标用户

坐在电脑前工作的知识工作者，尤其是开发者。

这些用户能理解：

- 我需要一个常驻本机、懂我工作上下文的助手
- 这个助手能帮我写代码、查资料、整理文档、执行命令
- 所有数据在我的电脑上，我能看到它做了什么
- 它越用越懂我的习惯和项目

## 当前产品形态

### 终端工作台

用户通过终端与 Smith 交互，两种入口：

1. **Ink 终端壳**（`smith`）— 富文本终端 UI，自动拉起后端，斜杠命令管理
2. **CLI 子命令**（`smith chat`、`smith sessions list`）— 可组合到脚本和工作流

### 交互模型

```
smith                           → 进入终端壳
smith chat -m "帮我分析这段代码"  → 单条消息
smith chat --session <id>       → 续聊已有会话
smith sessions list             → 查看历史会话
smith agent ensure              → 确保 Smith 已创建
```

## 产品支柱

1. **本地优先** — 数据在 `~/.agent-smith/`，引擎在本机，无需云端账号
2. **终端原生** — 终端是第一交互面，CLI 命令可组合到脚本中
3. **越用越聪明** — 记忆积累 + Dream 整理 + 偏好自动学习
4. **任务驱动** — 不是闲聊工具，是执行工具（DAG 技能链 + ReAct + 安全护栏）
5. **单 Agent 扩展** — 工具、技能、知识三层扩展，不增加 Agent 复杂度

## 核心场景

### 场景 A：日常开发辅助

1. 用户在终端输入 `smith`，进入工作台
2. 输入「帮我看看这个函数的性能问题」
3. Smith 的 task_router 判断为 debug 任务 → 走 sde-debug 技能链
4. ReAct 循环中读文件、分析代码、输出优化建议
5. 记忆中记录此次分析

### 场景 B：持续上下文

1. 用户昨天让 Smith 分析了一个 bug
2. 今天输入「昨天那个 bug 修好了吗？」
3. Smith 从记忆中检索昨天的会话上下文
4. 继续跟进，不用重复描述问题

### 场景 C：跨领域任务

1. 用户输入「帮我把这个 API 改一下，前端也调一下」
2. Smith 一次处理，注入前后端相关知识
3. 输出完整方案，不需要在两个"专家"之间切换

## MVP 边界

### 必须做

- Smith CLI + Ink 终端壳完整可用
- 技能系统 6 个内置技能
- 会话管理（创建、续聊、列表、查看）
- 记忆积累 + Dream 整理
- 安全护栏（24 条规则）

### 明确不做

- 不做多 Agent / 子 Agent
- 不做 GUI 桌面应用
- 不做技能市场
- 不做 IM 桥接
- 不做移动端

## 后续能力

- 知识注入系统（领域参考文档管理 + 自动检索注入）
- 更多内置技能（覆盖更多任务类型）
- 技能市场（社区共享 SKILL.md）
- 记忆压缩和长期事实库
- IM 桥接（飞书 / 钉钉）
