# 09 · 竞品架构对比

## 概述

本文档对 Agent-Smith 与三个主要竞品进行系统级架构对比分析。目标不是简单罗列功能差异，而是从执行模型、工具系统、记忆管线、安全模型、Prompt 工程、可扩展性六个核心维度，剖析各产品的设计取舍与技术权衡。

**对比对象：**

| 产品 | 定位 | 核心语言 | GitHub Stars |
|---|---|---|---|
| **Agent-Smith** | 本地优先个人助手 Agent 工作台，终端交付 | Python + TypeScript | — |
| **Claude Code** (Anthropic) | Terminal-native agentic coding assistant | TypeScript (98.4% 基础设施) | — |
| **Hermes Agent** (Nous Research) | 持久化 daemon Agent，研究级 RL 闭环 | Python | ~140K |
| **OpenClaw** (Peter Steinberger) | 跨平台 Agent 网关，marketplace 生态 | TypeScript | ~346K |

**为什么选这三个：**

- **Claude Code** — 同为 terminal-native Agent，但选择了截然不同的执行模型（纯 LLM 循环 vs. DAG+ReAct 混合），是最直接的架构对照。
- **Hermes Agent** — 同样使用 ReAct 循环，但引入了子 Agent 委派和 RL 训练闭环，代表"从 Agent 端优化 LLM 行为"的路线。
- **OpenClaw** — 网关 daemon 模型 + marketplace 生态，代表"平台化、多渠道、社区驱动"的路线。

## 架构模式对比

### 总览

四个产品在架构层面代表了四种不同的 Agent 设计哲学：

```
Agent-Smith:   确定性控制 + 受限 LLM 自由度
               task_router → skill_chain (DAG) → agent_loop (ReAct) → tool_guard

Claude Code:   最大化 LLM 自由度 + 基础设施兜底
               queryLoop() AsyncGenerator → tool dispatch → permission check

Hermes Agent:  预算约束 ReAct + 有限委派
               parent ReAct (budget=90) → child ReAct (budget=50, max=3)

OpenClaw:      控制面与执行面分离
               Gateway daemon → execution worker → platform adapter
```

### 架构模式对比矩阵

| 维度 | Agent-Smith | Claude Code | Hermes Agent | OpenClaw |
|---|---|---|---|---|
| **执行模型** | DAG + ReAct 混合 | 自由 LLM 循环 | ReAct + 子 Agent 委派 | Gateway + Worker 分离 |
| **任务分解** | 规则引擎路由 + DAG 编排 | LLM 自主判断 | LLM 判断 + 迭代预算 | LLM 判断 + 平台适配 |
| **质量控制** | 13 种门禁实现 | LLM 自检 + 用户确认 | 迭代预算硬限 | 社区评分 + 审核 |
| **确定性程度** | 高（代码管流程） | 低（LLM 全权） | 中（预算约束） | 中（网关路由确定） |
| **Agent 拓扑** | 单 Agent | 单 Agent + 子 Agent spawn | 父子 Agent (depth=1) | 单 Agent + 平台适配 |
| **常驻模式** | 按需启动 | 按需启动 | 持久 daemon | 持久 daemon |

### 设计哲学分析

**Agent-Smith 的核心赌注**是"宏观确定性控制，微观 LLM 自由度"。task_router 用规则引擎做第一层分流（不靠 LLM），skill_chain 用 DAG 强制步骤顺序和门禁（不靠 LLM），只在 agent_loop 的 ReAct 循环里给 LLM 自由度。这意味着 Agent 的行为路径是可预测、可审计的。代价是灵活性——新的任务模式需要显式定义 skill。

**Claude Code 的核心赌注**是"LLM 足够聪明，不需要 DAG"。整个执行就是一个 `queryLoop()` AsyncGenerator 循环：LLM 思考 → 调用工具 → 观察结果 → 继续思考。没有 skill chain，没有 DAG，没有门禁。质量依赖 LLM 本身的推理能力和 system prompt 的引导。98.4% 的代码量花在基础设施（工具实现、权限系统、上下文管理）上，只有 1.6% 是 AI 逻辑。这是一个"基础设施重、AI 逻辑轻"的架构。

**Hermes Agent 的核心赌注**是"Agent 可以通过训练变得更好"。ReAct 循环本身并不特殊，但 Hermes 加了两层：迭代预算（父 Agent 90 次、子 Agent 50 次的硬限）和 RL 闭环（Atropos 训练框架）。前者防止无限循环，后者让 Agent 从错误中学习。这是目前唯一有训练闭环的竞品。

**OpenClaw 的核心赌注**是"生态 > 单点能力"。Gateway daemon 做控制面（路由、权限、会话管理），execution worker 做执行面，platform adapter 做渠道适配。这种分离让 OpenClaw 能同时对接 20+ 个消息平台。技能不是内置的，而是从 ClawHub marketplace（13,000+ skills）按需加载。

## 执行模型

### 任务分解方式

| 产品 | 分解机制 | 分解深度 | 回溯策略 |
|---|---|---|---|
| **Agent-Smith** | task_router 规则引擎 → skill_chain DAG | 技能节点级（每个 skill 是一个 DAG 节点） | 门禁失败重试当前节点 |
| **Claude Code** | LLM 自主判断，无显式分解 | 无固定深度，LLM 自由决定 | LLM 自主决定是否重试 |
| **Hermes Agent** | LLM 判断 + 子 Agent 委派 | 最多 2 层（parent + child） | 迭代预算耗尽后失败 |
| **OpenClaw** | LLM 判断 + 技能路由 | 单层（一个技能处理一个任务） | 技能失败后回退到通用模式 |

### Agent-Smith 的 DAG + ReAct 混合执行

Agent-Smith 的执行管线是四层：

```
用户输入
  │
  ▼
task_router (规则引擎)
  │  基于关键词/模式匹配判断任务类型
  │  不调用 LLM，确定性路由
  ▼
skill_chain (DAG 编排)
  │  按技能定义的步骤顺序执行
  │  每个节点有入口门禁 + 出口门禁
  │  13 种门禁实现（format_check, test_pass, lint_clean 等）
  ▼
agent_loop (ReAct 循环)
  │  思考 → 工具调用 → 观察 → 思考...
  │  LLM 在此获得"做事"的自由度
  ▼
tool_guard (安全层)
     29 条安全规则，9 个类别
     regex 匹配，拦截危险操作
```

13 种门禁实现覆盖：格式检查、测试通过、lint 通过、类型检查、安全扫描、覆盖率阈值、性能基准、人工审批、依赖审计、文档完整性、向后兼容、变更集大小、CI 通过。这些门禁是确定性的——不靠 LLM 判断，靠代码检查。

### Claude Code 的自由循环

Claude Code 的整个执行就是 `queryLoop()`：

```typescript
async function* queryLoop(messages, tools, config) {
  while (true) {
    const response = await llm.complete(messages, tools);
    yield response;
    if (response.stop_reason === 'end_turn') break;
    // 处理工具调用，结果加入 messages
    for (const toolUse of response.tool_calls) {
      const result = await executeTool(toolUse);
      messages.push(result);
    }
  }
}
```

没有 DAG，没有门禁，没有规则引擎。LLM 决定调用什么工具、以什么顺序、是否需要重试。质量控制完全依赖：
1. System prompt 的引导（27 个 lifecycle hook 可注入额外指令）
2. 工具本身的校验（如 Edit 工具会验证 old_string 唯一性）
3. 用户的权限确认（allowlist + ask + deny 模型）

这种模式的优势是灵活——任何新任务都不需要定义 skill，LLM 自行推理。劣势是不可预测——同样的任务，两次执行可能走完全不同的路径。

### Hermes Agent 的预算 ReAct

Hermes 的执行模型核心是"带预算的 ReAct"：

```
Parent Agent (iteration_budget = 90)
  │
  ├── 自行处理（多数任务）
  │
  └── 委派给 Child Agent (iteration_budget = 50)
       ├── 最多 3 个并发子 Agent
       └── 委派深度限制 = 1（子 Agent 不能再委派）
```

迭代预算是硬性约束：父 Agent 最多思考-行动 90 个回合，子 Agent 50 个回合。预算耗尽即失败，不会无限循环。这比 Agent-Smith 的门禁更粗粒度（整体次数限制 vs. 每步质量检查），但实现更简单。

### 执行模型评估

| 评估维度 | Agent-Smith | Claude Code | Hermes | OpenClaw |
|---|---|---|---|---|
| **可预测性** | 高 | 低 | 中 | 中 |
| **灵活性** | 中 | 高 | 中 | 高（靠生态） |
| **新任务适应** | 需定义 skill | 零配置 | 零配置 | 需安装 skill |
| **失败可控性** | 高（门禁拦截） | 低（LLM 自行判断） | 中（预算硬限） | 中（技能隔离） |
| **调试友好性** | 高（DAG 路径可追踪） | 中（有日志但路径不固定） | 中（预算消耗可见） | 低（跨平台链路长） |

## 工具系统

### 工具规模对比

| 产品 | 内置工具数 | 工具分类 | 外部扩展机制 |
|---|---|---|---|
| **Agent-Smith** | 14 providers | 文件、Git、Web、搜索、Shell、记忆、编辑 | 工具 provider Python 类 |
| **Claude Code** | 54 tools | 文件、编辑、搜索、Git、浏览器、Agent spawn、Notebook | MCP (Model Context Protocol) |
| **Hermes Agent** | ~30 tools | 文件、Shell、Web、代码分析、子 Agent 管理 | 技能学习自动生成工具 |
| **OpenClaw** | ~20 core tools | 文件、Web、消息、日历、邮件 | ClawHub marketplace |

### MCP 集成

MCP (Model Context Protocol) 是 Anthropic 推出的外部工具集成标准。当前各产品的 MCP 支持状态：

| 产品 | MCP 支持 | 实现方式 |
|---|---|---|
| **Agent-Smith** | 未实现 | — |
| **Claude Code** | 原生支持 | 配置 `mcpServers`，自动发现工具 + resources + prompts |
| **Hermes Agent** | 部分支持 | 通过 MCP adapter 桥接 |
| **OpenClaw** | 原生支持 | Gateway 层统一管理 MCP server 生命周期 |

Agent-Smith 缺失 MCP 支持是一个显著的生态短板。MCP 已成为 Agent 工具互操作的事实标准，Claude Code 和 OpenClaw 的工具生态都建立在 MCP 之上。

### 工具调用安全

| 产品 | 安全模型 | 实现 |
|---|---|---|
| **Agent-Smith** | 29 条 regex 规则 / 9 类 | tool_guard 层，规则引擎，匹配即拦截 |
| **Claude Code** | 7 层安全模型 | ML 两阶段分类器（0.4% 误报率）+ allowlist/ask/deny |
| **Hermes Agent** | 7 层纵深防御 | 沙箱 + 权限 + 审计 |
| **OpenClaw** | 平台级 ACL | Gateway 层权限控制 + 技能签名验证 |

Claude Code 的安全模型明显更成熟。ML 两阶段分类器先用轻量模型快速判断（<10ms），再用完整模型精确分类，误报率压到 0.4%。相比之下，Agent-Smith 的 regex 规则是"匹配即拦截"，存在误报（合法命令被规则误匹配）和漏报（规则未覆盖的危险操作）两个方向的风险。

### 懒加载策略

| 产品 | 工具加载策略 | token 节省 |
|---|---|---|
| **Agent-Smith** | 全量加载 14 个 provider 的工具定义 | 无 |
| **Claude Code** | 54 个工具全量注入 system prompt | 无（依赖 prefix cache 抵消） |
| **Hermes Agent** | 按任务类型加载相关工具子集 | 中等 |
| **OpenClaw** | marketplace 技能按需拉取，用完卸载 | 高 |

## 记忆与持久化

### 记忆架构对比

| 维度 | Agent-Smith | Claude Code | Hermes Agent | OpenClaw |
|---|---|---|---|---|
| **会话记忆** | SQLite + recent.jsonl | 对话上下文窗口 | 对话上下文 + 摘要 | 对话上下文 + 摘要 |
| **跨会话持久化** | durable.md + episodes.jsonl | CLAUDE.md + auto-memory 文件 | 持久化知识库 | sqlite-vec 向量嵌入 |
| **记忆编译** | recent → compile → durable 管线 | 无编译，直接文件读写 | 5 阶段自主学习 | 无编译 |
| **记忆巩固** | Dream 机制（LLM 合并） | 无 | RL 训练闭环 (Atropos) | 无 |
| **搜索** | FTS5 + sqlite-vec 混合 | 文件系统 grep | 向量搜索 | sqlite-vec 向量搜索 |
| **用户偏好** | UserPreferenceLearner (4 维) | CLAUDE.md 手动维护 | 自主学习 | 无 |

### Agent-Smith 的三类语义记忆管线

Agent-Smith 的记忆系统是当前最结构化的设计：

```
对话交互
  │
  ▼
recent.jsonl          ← 原始对话片段（时间序列）
  │
  ▼ compile (定时触发)
  │
durable.md            ← 编译后的持久知识（去重、合并、结构化）
  │
  ▼ Dream (低活跃期触发)
  │
durable.md (更新)     ← LLM 巩固后的知识（交叉引用、矛盾消解）
```

加上 episodes.jsonl（事件片段）和 context.md（交互配置），形成三类语义记忆：
- **recent** — 短期，原始，时间有序
- **durable** — 长期，编译后，语义结构化
- **episodes** — 事件片段，叙事形式，用于经验回顾

搜索使用 FTS5（全文检索）+ sqlite-vec（向量相似度）混合策略。FTS5 处理精确关键词匹配，sqlite-vec 处理语义相似度。UserPreferenceLearner 从交互中提取 4 个维度的偏好：沟通风格、技术偏好、工作习惯、领域知识。

### Claude Code 的极简记忆

Claude Code 的记忆模型极其简洁：

- **CLAUDE.md** — 项目级的持久化指令文件，用户手动或 auto-memory 自动维护
- **Auto-memory** — 从对话中提取的事实，写入 `~/.claude/projects/<hash>/memory/` 下的 .md 文件
- **无编译、无巩固** — 文件就是记忆，grep 就是搜索

这种设计的优势是透明——用户可以直接阅读和编辑记忆文件。劣势是缺乏结构——随着记忆积累，文件会变得冗长，没有自动去重和合并机制。

### Hermes Agent 的 5 阶段学习

Hermes 的记忆不仅是存储，还包含学习闭环：

```
1. 观察    — 记录 Agent 行为和结果
2. 提取    — 从观察中提取模式和规则
3. 验证    — 在新场景中测试提取的规则
4. 内化    — 通过 RL (Atropos) 训练更新模型权重
5. 应用    — 在后续任务中使用学到的能力
```

这是唯一一个将记忆与模型训练直接打通的架构。Agent-Smith 的 Dream 机制（LLM 合并 durable.md）只是推理时的知识重组，不改变模型本身。Hermes 的 Atropos 训练框架则实际更新模型权重，让 Agent 从错误中"真正学到"。

### 上下文压缩

| 产品 | 压缩策略 | 压缩层数 |
|---|---|---|
| **Agent-Smith** | 无压缩（依赖 cut_order 裁剪 prompt 层） | 0 |
| **Claude Code** | 5 层压缩管线 | 5（摘要→截断→丢弃→压缩→再摘要） |
| **Hermes Agent** | 上下文窗口摘要 | 1 |
| **OpenClaw** | 向量化摘要 | 1 |

Agent-Smith 缺乏上下文压缩是长对话场景的短板。Claude Code 的 5 层压缩管线能在 context window 逼近上限时自动摘要历史，保持对话连贯性。Agent-Smith 的 `cut_order` 机制只能裁剪 prompt 的各层（先砍优先级低的层），但不压缩对话历史本身。

## 安全模型

### 安全层级对比

| 层级 | Agent-Smith | Claude Code | Hermes Agent |
|---|---|---|---|
| **输入校验** | tool_guard regex 匹配 | ML 分类器第一阶段 | 输入清洗层 |
| **权限控制** | 9 类安全规则 | allowlist / ask / deny per tool | 角色基权限 |
| **执行沙箱** | 无（直接执行） | 无（直接执行，靠权限拦截） | 容器沙箱 |
| **输出审计** | 无 | 工具结果审计 | 输出过滤层 |
| **路径保护** | regex 路径检查 | ML 分类器 + 路径白名单 | 路径沙箱 |
| **密钥保护** | regex 模式检查 | ML 检测 + .env 文件保护 | 密钥管理器 |
| **网络控制** | URL regex 规则 | 域名白名单 | 出口防火墙 |

### Agent-Smith 安全规则详情

29 条规则分布在 9 个类别中：

| 类别 | 规则数 | 示例 |
|---|---|---|
| 文件系统保护 | 5 | 禁止 rm -rf /、禁止写入 /etc |
| Git 安全 | 4 | 禁止 force push、禁止 reset --hard |
| 网络安全 | 3 | 禁止内网 IP 访问、禁止非 HTTPS |
| 密钥保护 | 3 | 检测 API key 模式、禁止 .env 读取 |
| 命令注入 | 4 | 禁止管道到 sh、禁止 eval |
| 资源限制 | 2 | 禁止 fork 炸弹、内存限制 |
| 数据安全 | 3 | 禁止 DROP TABLE、禁止 DELETE 无 WHERE |
| 进程安全 | 2 | 禁止 kill -9、禁止修改 crontab |
| 输出安全 | 3 | 禁止输出中包含密钥、token 清洗 |

### Claude Code 的 ML 安全模型

Claude Code 的安全模型是竞品中最成熟的：

```
用户请求
  │
  ▼
第一阶段 ML 分类器 (<10ms)
  │  轻量模型，快速判断：安全 / 需要检查 / 危险
  │
  ├── 安全 → 直接执行
  ├── 需要检查 → 第二阶段
  └── 危险 → 直接拦截
        │
        ▼
第二阶段 ML 分类器 (~50ms)
  │  完整模型，精确分类
  │  综合考虑：命令内容 + 上下文 + 用户历史 + 项目配置
  │
  ├── 允许 → 执行
  ├── 询问 → 弹出权限确认
  └── 拒绝 → 拦截 + 解释原因
```

**0.4% 误报率**意味着每 250 次合法操作只有 1 次被错误拦截。相比之下，Agent-Smith 的 regex 规则没有公开的误报率数据，但根据规则的粗粒度特性（如"包含 rm -rf 就拦截"），误报率预计显著更高。

### 安全模型评估

| 评估维度 | Agent-Smith | Claude Code | Hermes | OpenClaw |
|---|---|---|---|---|
| **覆盖面** | 中（29 条规则） | 高（ML + 规则双层） | 高（7 层纵深） | 中（平台级 ACL） |
| **精确度** | 低（regex 粗匹配） | 高（0.4% 误报） | 中（沙箱可能过严） | 中（ACL 粗粒度） |
| **适应性** | 低（需手动加规则） | 高（ML 自动学习） | 中（可配置） | 低（需手动配置） |
| **透明度** | 高（规则可读） | 中（ML 黑盒） | 低（沙箱复杂） | 中（ACL 可读） |
| **绕过风险** | 高（regex 易绕过） | 低（ML 泛化强） | 低（沙箱隔离） | 中（ACL 绕过可能） |

## Prompt 工程

### System Prompt 分层对比

| 产品 | 层数 | 结构 | 缓存策略 | Token 预算管理 |
|---|---|---|---|---|
| **Agent-Smith** | 9 层 | role / style / workflow / toolbox / knowledge / context / skill / safety / memory | prefix cache hints + cut_order | cut_order 按优先级裁剪层 |
| **Claude Code** | ~5 层 | system / tools / CLAUDE.md / hooks / conversation | Anthropic prefix caching | 5 层压缩管线 |
| **Hermes Agent** | 3 层 | Stable / Context / Volatile | 三级缓存（稳定层最大化命中） | 迭代预算间接限制 |
| **OpenClaw** | 2 层 | SOUL.md persona / tool context | persona 层缓存 | 无显式管理 |

### Agent-Smith 的 9 层 Prompt 组装

Agent-Smith 的 prompt assembler 是所有竞品中最精细的分层设计：

```
Layer 0: role.md          ← 身份定义（"你是 Smith"）
Layer 1: style.md         ← 输出风格（语气、格式偏好）
Layer 2: workflow.md      ← 当前工作流步骤
Layer 3: toolbox.md       ← 可用工具清单和使用规范
Layer 4: knowledge/       ← 按需注入的领域知识文档
Layer 5: context.md       ← 用户交互配置和偏好
Layer 6: skill/           ← 当前激活技能的指令
Layer 7: safety rules     ← 安全规则注入
Layer 8: memory           ← durable.md + recent 摘要
```

每一层有 `cut_order` 优先级。当总 token 超过预算时，assembler 从 `cut_order` 最高（优先级最低）的层开始裁剪。例如 knowledge 层通常 `cut_order` 较高，会先于 role/style 被裁剪。

Prefix cache hints 标记哪些层在多轮对话中不变（如 role.md、style.md），让 LLM provider 的 prefix cache 命中率最大化。

### Hermes 的三级缓存策略

Hermes 将 prompt 分为三个缓存层级：

| 层级 | 内容 | 变化频率 | 缓存策略 |
|---|---|---|---|
| **Stable** | 系统人设、基础工具定义、安全规则 | 跨会话不变 | 长期缓存，最大化 prefix cache 命中 |
| **Context** | 任务上下文、相关文件、知识检索结果 | 每个任务变化 | 中期缓存，按任务切换刷新 |
| **Volatile** | 对话历史、工具调用结果、中间状态 | 每轮变化 | 不缓存 |

这种分层比 Agent-Smith 的 9 层更粗粒度，但缓存效率可能更高——因为层数少意味着 Stable 层的 token 占比更大。

### Claude Code 的 Lifecycle Hooks

Claude Code 有 27 个 lifecycle hook 事件，允许在 prompt 的各个阶段注入自定义内容：

| Hook 类别 | 事件数 | 示例 |
|---|---|---|
| 会话生命周期 | 4 | session_start, session_end, context_restore |
| 工具调用 | 8 | pre_tool_call, post_tool_call, tool_error |
| 消息处理 | 6 | pre_message, post_message, stream_chunk |
| 文件操作 | 5 | pre_edit, post_edit, pre_write |
| 权限 | 4 | permission_request, permission_grant |

Agent-Smith 没有 lifecycle hook 系统。技能的 SKILL.md 定义了工作流步骤，但不能在任意时机注入逻辑。这限制了第三方对 Agent 行为的定制能力。

### Agent-Smith 的 File-Based Persona

Agent-Smith 用 6 个文件定义 Agent 人设：

```
agents/smith/
├── role.md        ← 身份和核心指令
├── style.md       ← 输出风格和语气
├── workflow.md    ← 默认工作流
├── toolbox.md     ← 工具使用规范
├── context.md     ← 交互上下文和偏好
└── config.yaml    ← 运行时配置参数
```

这比 Claude Code 的单文件 CLAUDE.md 更结构化，比 OpenClaw 的 SOUL.md 更精细。优势是每个维度可以独立修改（改风格不影响角色）；劣势是文件散落，新用户理解成本高。

### OpenClaw 的 SOUL.md + ClawHub

OpenClaw 的可扩展性是竞品中最强的：

- **SOUL.md** — persona 定义文件，支持工具调用级别的人设包装（工具调用时也保持人设一致性）
- **ClawHub marketplace** — 13,000+ 技能，社区贡献 + 审核机制
- **20+ 平台适配** — 同一个 Agent 可以同时服务 Slack、Discord、微信、Telegram 等

但这种"平台化"策略也带来复杂性。每个平台的消息格式、权限模型、用户体验差异巨大，Gateway 层需要大量适配逻辑。Agent-Smith 选择"终端 only"反而让交付体验更一致。

## Agent-Smith 差异化优势

### 1. 唯一的 DAG + ReAct 混合执行模型

Agent-Smith 是四个产品中唯一将确定性编排（DAG）与 LLM 推理（ReAct）结合的架构。这不是简单的叠加——DAG 管控的是"做什么步骤、以什么顺序"，ReAct 管控的是"每个步骤里怎么做"。门禁系统在两者之间插入质量检查点。

竞品要么完全信任 LLM（Claude Code），要么用粗粒度预算限制（Hermes），要么不介入执行细节（OpenClaw）。Agent-Smith 的混合模型在可预测性和灵活性之间找到了独特的平衡点。

### 2. 13 种门禁实现的精细质量控制

没有其他竞品有等价的门禁系统。Claude Code 依赖 LLM 自检，Hermes 依赖迭代预算，OpenClaw 依赖社区评分。Agent-Smith 的门禁是代码级的确定性检查——测试必须通过、lint 必须干净、覆盖率必须达标——不给 LLM "差不多就行"的余地。

### 3. 三类语义记忆 + Dream 巩固管线

Agent-Smith 的 recent → compile → durable 管线和 Dream 巩固机制是最结构化的记忆系统。Claude Code 只有文件读写，Hermes 有学习闭环但不分层，OpenClaw 只有向量存储。Agent-Smith 的三类语义记忆（recent/durable/episodes）各有明确职责，Dream 机制在低活跃期自动合并和去重。

### 4. 9 层 Prompt 组装 + cut_order 预算管理

9 层 prompt 组装是最精细的分层，cut_order 机制提供了优雅的 token 预算管理——在空间不足时按优先级裁剪，而不是粗暴截断。这比 Claude Code 的压缩管线更可控（知道砍的是什么），比 Hermes 的三级缓存更精细（9 层 vs. 3 层）。

### 5. File-based Persona 的结构化人设

6 个文件分离人设的不同维度，比 CLAUDE.md（单文件混杂）和 SOUL.md（单文件）更具可维护性。改风格不影响角色，改工具不影响工作流，每个维度独立演进。

## 借鉴建议

按优先级从高到低排列。每条标注预期收益和实现复杂度。

### P0 — 生态卡位

#### 1. 实现 MCP 支持

- **借鉴对象**：Claude Code、OpenClaw
- **现状**：Agent-Smith 的 14 个内置工具是封闭系统，无法接入外部 MCP server
- **收益**：接入 MCP 生态意味着立即获得数千个外部工具（数据库、API、文件系统、云服务），无需逐个实现
- **建议**：在 engine 层实现 MCP client，在 tool_guard 层统一安全检查
- **复杂度**：中（协议已标准化，主要工作是 client 实现 + tool_guard 集成）

### P1 — 核心能力补齐

#### 2. 增加上下文压缩管线

- **借鉴对象**：Claude Code（5 层压缩）
- **现状**：长对话只能靠 cut_order 裁剪 prompt 层，不压缩对话历史
- **收益**：支持长会话不丢失关键上下文；当前模式下长对话会导致 prompt 层被裁剪
- **建议**：在 agent_loop 中增加对话历史摘要机制，与 cut_order 互补（cut_order 管 prompt 层，压缩管对话历史）
- **复杂度**：中

#### 3. 引入 Lifecycle Hook 系统

- **借鉴对象**：Claude Code（27 事件）
- **现状**：技能只能通过 SKILL.md 定义工作流步骤，无法在任意时机注入逻辑
- **收益**：开放 hook 让第三方在 pre_tool_call、post_edit、session_start 等时机注入自定义逻辑，大幅提升可扩展性
- **建议**：在 agent_loop 的关键节点注入 hook dispatcher，hook 定义放在 SKILL.md 或独立配置中
- **复杂度**：中

### P2 — 安全强化

#### 4. 升级安全模型：从 regex 到 ML 分类

- **借鉴对象**：Claude Code（ML 两阶段分类器，0.4% 误报）
- **现状**：29 条 regex 规则，粗粒度匹配，误报率高，易被绕过
- **收益**：降低误报率，提升绕过难度，减少用户被误拦截的挫败感
- **建议**：短期可增加规则的上下文敏感性（不只看命令本身，还看执行目录和前序操作）；中期可引入轻量 ML 分类器做第一阶段快速判断
- **复杂度**：高（ML 分类器需要训练数据和持续维护）

### P3 — 差异化增强

#### 5. 引入 RL 训练闭环

- **借鉴对象**：Hermes Agent（Atropos）
- **现状**：Dream 机制只做推理时的知识重组，不改变模型行为
- **收益**：让 Agent 从错误中真正学习，而不只是记住知识
- **建议**：这是长期方向。短期可先做"行为评分 + 偏好数据积累"，为未来 RL 训练准备数据集
- **复杂度**：极高（需要自有模型或 fine-tuning 能力）

#### 6. 技能 Marketplace

- **借鉴对象**：OpenClaw（ClawHub，13,000+ skills）
- **现状**：28 个内置技能，无分发和安装机制
- **收益**：社区驱动的技能增长，减轻核心团队的维护负担
- **建议**：先做技能的标准化打包格式（SKILL.md + 依赖声明 + 签名验证），再做注册和发现
- **复杂度**：高（需要注册中心、审核流程、版本管理）

### 总结优先级矩阵

| 优先级 | 建议 | 借鉴对象 | 收益 | 复杂度 |
|---|---|---|---|---|
| **P0** | MCP 支持 | Claude Code / OpenClaw | 生态接入 | 中 |
| **P1** | 上下文压缩 | Claude Code | 长会话能力 | 中 |
| **P1** | Lifecycle Hooks | Claude Code | 可扩展性 | 中 |
| **P2** | ML 安全分类 | Claude Code | 安全精度 | 高 |
| **P3** | RL 训练闭环 | Hermes | 自主学习 | 极高 |
| **P3** | 技能 Marketplace | OpenClaw | 社区生态 | 高 |
