# 工作流规范

## 任务评估

收到任务后首先评估类型，选择对应路由：

| 任务类型 | 路由 | 触发条件 |
|---------|------|---------|
| Bug Fix | 调试路由 | 数据异常、报表错误、ETL 失败、指标偏差 |
| Feature | 功能路由 | 新报表、新指标、数据管道、分析模型 |
| Refactor | 重构路由 | 查询优化、数据模型重构、管道重组 |
| 简单问答 | 直接回复 | 不涉及代码变更的咨询 |

评估维度：
- 数据范围：单表 / 跨表 / 跨数据源
- 时效要求：实时 / 准实时 / 批量
- 是否需要数据回填或历史数据处理

## 强制技能链（不可跳过）

### Feature 路由
```
planning → architecture(仅大型变更) → testing-strategy → change-validation → code-review
```

### Bug Fix 路由
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### Refactor 路由
```
planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix **必须**使用 sde-debug 和 testing-strategy，不可跳过
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review **始终**是最后一步，不可跳过
- testing-strategy 必须在 change-validation 之前完成

## 公共步骤

### 1. 理解 (Understand)

- 明确分析目标和业务问题
- 确认数据源、时间范围、粒度要求
- 识别关键指标定义和计算口径
- 确认受众：技术团队 / 业务方 / 管理层
- 明确交付形式：报表 / 仪表盘 / 临时分析 / 数据管道

→ **理解门禁**：能准确复述分析目标 + 指标口径已确认 + 数据源已明确

### 2. 调查 (Investigate)

- 探查数据源质量：完整性、一致性、时效性
- 检查现有数据模型和 ETL 管道
- 确认数据权限和脱敏要求
- 分析数据分布和异常值
- 确认现有报表是否可复用

调查方法（数据分析特定）：
- 执行数据剖析（profiling）：空值率、唯一值、分布
- 检查 ETL 日志和数据刷新时间
- 验证指标计算逻辑与业务方理解一致
- 抽样对比不同数据源的一致性

→ **调查门禁**：数据质量已评估 + 技术方案可行 + 依赖已明确

### 3. 实现 (Implement)

- 编写数据提取和清洗逻辑
- 实现指标计算和聚合管道
- 构建数据模型（维度表 / 事实表 / 宽表）
- 编写数据质量校验规则
- 构建可视化报表或仪表盘
- 编写数据管道的单元测试

实现规范：
- SQL 可读性优先：CTE 分层、有意义的别名
- 计算口径必须有注释说明
- 大查询必须有性能评估（EXPLAIN）
- 数据管道必须有幂等设计
- 敏感数据必须脱敏处理

→ **契约对齐门禁**：实现方案与规划一致 + 指标口径正确 + 查询性能达标

### 4. 验证 (Verify)

- 数据准确性：与已知基准对比交叉验证
- 数据完整性：检查空值、重复、缺失
- 指标合理性：同比/环比趋势是否合理
- 性能验证：查询执行时间在可接受范围
- 可视化验证：图表是否准确反映数据
- 边界测试：空数据、极端值、跨时区

→ **验证门禁**：数据交叉验证通过 + 查询性能达标 + 无数据质量问题

### 5. 交付 (Deliver)

- 附数据口径说明和计算逻辑文档
- 标注数据刷新频率和延迟
- 提供数据质量监控方案
- 标注已知局限和注意事项
- 交付可复现的分析脚本（非截图）

→ **交付门禁**：数据已验证 + 口径文档已附 + 可复现

## 自主执行策略

默认自主执行。仅在以下情况询问用户：
1. 需求有多种互斥解读，无法自行判断
2. 需要访问新的数据源或提升数据权限
3. 指标口径有歧义，业务方理解不一致

反模式（明确禁止）：
- "要我继续吗？" → 默认继续
- "要我创建文件吗？" → 直接创建
- "要我运行查询吗？" → 直接运行
- "要我生成报表吗？" → 直接生成
- "这样可以吗？" → 做出判断，执行最佳方案

## 回退策略

```
Implementation 失败 → 回退到 Planning，重新评估方案
Verify 失败 → 回退到 Implementation，修复后重新验证
同一失败出现 2 次 → 换策略（不同查询方案/数据源）
换策略后仍失败 → 上报用户，附带已尝试方案和失败原因
```

## 跨领域原则

- **指标口径一致性**：同一指标全组织统一定义
- **数据质量优先**：垃圾进垃圾出，质量校验前置
- **可复现性**：分析结果必须可通过脚本复现
- **性能意识**：大数据量查询必须优化，避免全表扫描
- **隐私合规**：PII 数据必须脱敏，访问必须有审计日志
- **版本管理**：SQL/ETL 脚本纳入版本控制
- **文档即交付**：口径说明和数据字典与代码同等重要

## 评估敏感信号检测

当检测到以下信号时，提升验证严格度：
- 涉及财务/营收数据
- 涉及用户隐私数据
- 数据将用于管理层决策
- 涉及合规报告或监管数据
- 跨数据源 JOIN 的一致性风险
