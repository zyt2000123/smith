# 工作流规范

## 任务评估

收到任务后首先评估类型，选择对应路由：

| 任务类型 | 路由 | 触发条件 |
|---------|------|---------|
| Bug Fix | 调试路由 | 部署失败、服务宕机、配置错误、监控告警 |
| Feature | 功能路由 | 新流水线、基础设施变更、监控增强 |
| Refactor | 重构路由 | IaC 重构、流水线优化、成本优化 |
| 简单问答 | 直接回复 | 不涉及基础设施变更的咨询 |

评估维度：
- 影响范围：单服务 / 多服务 / 全集群 / 跨环境
- 风险等级：低（监控变更）/ 中（流水线调整）/ 高（生产环境变更）
- 是否需要停机窗口或回滚计划

## 强制技能链（不可跳过）

### Feature 路由
```
planning → architecture(仅大型变更) → testing-strategy → change-validation → code-review
```

### Bug Fix 路由
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### Refactor 路由
```
planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix **必须**使用 sde-debug 和 testing-strategy，不可跳过
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review **始终**是最后一步，不可跳过
- testing-strategy 必须在 change-validation 之前完成

## 公共步骤

### 1. 理解 (Understand)

- 明确基础设施变更的目标和约束
- 确认目标环境：开发 / 预发布 / 生产
- 识别依赖：网络、存储、密钥、外部服务
- 确认 SLA/SLO 要求和停机容忍度
- 评估成本影响

→ **理解门禁**：能准确复述需求 + 识别至少 3 个风险点 + 明确回滚路径

### 2. 调查 (Investigate)

- 审查现有基础设施配置（IaC 代码、配置文件）
- 检查当前监控和告警规则
- 确认 CI/CD 流水线现状
- 审查密钥和凭证管理方式
- 检查资源使用率和成本趋势

调查方法（DevOps 特定）：
- 检查 Terraform/Pulumi state 和 plan output
- 审查 Kubernetes manifests 和 Helm charts
- 检查 CI/CD 流水线日志和执行历史
- 审查监控仪表盘和告警历史

→ **调查门禁**：已确认现有状态 + 技术方案可行 + 回滚计划就绪

### 3. 实现 (Implement)

- IaC 优先：所有基础设施变更通过代码管理
- 先在开发环境验证，再逐步推进到生产
- 配置变更使用渐进式发布（canary / blue-green）
- 添加/更新监控告警规则
- 编写自动化测试：IaC 验证、冒烟测试、集成测试
- 更新运维手册和故障排查文档

实现规范：
- 基础设施即代码：不手动修改生产配置
- 密钥管理：使用 Vault/KMS，不在代码中硬编码
- 最小权限：IAM 策略遵循最小权限原则
- 不可变基础设施：替换而非修改
- 所有变更必须有回滚脚本

→ **契约对齐门禁**：实现方案与规划一致 + IaC plan 无意外 + lint 零错误

### 4. 验证 (Verify)

- 在 staging 环境执行完整部署流程
- 验证监控告警是否按预期触发
- 执行回滚演练，确认可快速恢复
- 检查安全扫描结果（容器镜像、依赖漏洞）
- 验证负载均衡和自动扩缩容
- 确认日志收集和可观测性完整
- 执行基本负载测试

→ **验证门禁**：staging 部署成功 + 监控就绪 + 回滚已验证 + 安全扫描通过

### 5. 交付 (Deliver)

- 更新运维手册和故障排查文档
- PR 包含 terraform plan / diff 输出
- 标注停机窗口和影响范围
- 提供回滚步骤和紧急联系人
- 更新架构图（如涉及拓扑变更）

→ **交付门禁**：staging 验证通过 + 运维文档已更新 + 回滚计划已确认

## 自主执行策略

默认自主执行。仅在以下情况询问用户：
1. 需求有多种互斥解读，无法自行判断
2. 生产环境变更（任何直接影响生产的操作）
3. 涉及成本显著增加的资源变更

反模式（明确禁止）：
- "要我继续吗？" → 默认继续
- "要我创建文件吗？" → 直接创建
- "要我运行测试吗？" → 直接运行
- "要我 apply 变更吗？" → 开发/staging 直接执行
- "这样可以吗？" → 做出判断，执行最佳方案

## 回退策略

```
Implementation 失败 → 回退到 Planning，重新评估方案
Verify 失败 → 回退到 Implementation，修复后重新验证
同一失败出现 2 次 → 换策略（不同架构/工具链）
换策略后仍失败 → 上报用户，附带已尝试方案和失败原因
```

## 跨领域原则

- **基础设施即代码**：不手动修改，所有变更可审计可复现
- **不可变基础设施**：替换而非修补，避免配置漂移
- **安全左移**：在 CI 阶段就运行安全扫描
- **可观测性三支柱**：日志 + 指标 + 追踪，缺一不可
- **成本意识**：每个资源都应有成本标签和预算告警
- **灾难恢复**：定期验证备份和恢复流程
- **最小权限**：IAM 策略按需授权，定期审计

## 评估敏感信号检测

当检测到以下信号时，提升验证严格度：
- 涉及生产环境变更
- 涉及网络安全组/防火墙规则
- 涉及密钥轮换或凭证管理
- 涉及数据备份/恢复策略
- 涉及多区域/多集群部署
