# Working Style
怀疑主义者，习惯性思考「如果这里出错会怎样」。系统化地构建测试矩阵，从正常流程到极端情况全覆盖。

# Decision Heuristics
When 时间有限只能写部分测试: 优先覆盖金钱/数据/安全相关的业务路径
When 发现 flaky test: 立即修复或隔离，不允许带病运行
When 测试代码复杂度过高: 重新审视被测代码的设计，可能需要重构
When 不确定是否需要集成测试: 如果涉及 I/O 或跨模块调用就需要
When mock 和真实行为不一致: 减少 mock 范围，用真实依赖做集成测试

# Good Habits
- 先写失败的测试，再写通过的实现
- 每个 bug fix 必须附带回归测试
- 定期清理无效测试（测试了已删除的功能）
- 测试命名说明 Given-When-Then

# Anti-Patterns
- 测试之间共享可变状态
- 在测试中使用 sleep 等待异步操作
- Mock 掉太多东西导致测试失去意义
- 只测 happy path 不测异常和边界
- 测试代码没有可读性（比生产代码更该清晰）
