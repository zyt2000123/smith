# 工作流规范

## 任务评估

收到任务后首先评估类型，选择对应路由：

| 任务类型 | 路由 | 触发条件 |
|---------|------|---------|
| Bug Fix | 调试路由 | 测试框架异常、误报/漏报、flaky test |
| Feature | 功能路由 | 新测试套件、测试基础设施、自动化流程 |
| Refactor | 重构路由 | 测试重构、提升覆盖率、减少冗余 |
| 简单问答 | 直接回复 | 不涉及代码变更的咨询 |

评估维度：
- 测试层级：单元 / 集成 / E2E / 性能 / 安全
- 影响范围：单模块 / 跨模块 / 全量回归
- 是否需要测试基础设施变更

## 强制技能链（不可跳过）

### Feature 路由
```
planning → architecture(仅大型变更) → testing-strategy → change-validation → code-review
```

### Bug Fix 路由
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### Refactor 路由
```
planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix **必须**使用 sde-debug 和 testing-strategy，不可跳过
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review **始终**是最后一步，不可跳过
- testing-strategy 必须在 change-validation 之前完成

## 公共步骤

### 1. 理解 (Understand)

- 明确被测功能的行为规格和验收标准
- 识别测试边界：正常路径、异常路径、边界值
- 确认测试环境要求和外部依赖
- 评估风险区域：高复杂度 / 频繁变更 / 历史 bug 密集区
- 确认覆盖率目标和非功能需求（性能、安全）

→ **理解门禁**：能准确复述验收标准 + 识别至少 5 个测试场景 + 明确测试层级

### 2. 调查 (Investigate)

- 审查现有测试套件，确认覆盖现状
- 检查 CI 流水线中的测试配置
- 分析历史 flaky test 和失败模式
- 确认 mock/stub/fixture 的可用性
- 审查代码覆盖率报告，识别盲区

调查方法（测试特定）：
- 运行现有测试套件，记录基线通过率
- 分析代码覆盖率报告（行覆盖 + 分支覆盖）
- 检查 CI 历史，识别 flaky test 模式
- 审查测试数据管理策略

→ **调查门禁**：已确认覆盖现状 + 技术方案可行 + 测试基础设施就绪

### 3. 实现 (Implement)

- 按测试金字塔原则分层编写
- 单元测试：隔离被测单元，mock 外部依赖
- 集成测试：验证模块间交互，使用真实依赖
- E2E 测试：覆盖关键用户流程
- 编写测试 fixture 和工厂方法
- 添加测试数据清理逻辑

实现规范：
- 测试命名：`test_<被测行为>_when_<条件>_should_<预期结果>`
- 每个测试只验证一个行为
- 避免测试间依赖和顺序耦合
- 测试数据自包含，不依赖外部状态
- flaky test 必须立即修复或隔离

→ **契约对齐门禁**：测试方案与策略一致 + 覆盖率达标 + 无 flaky test

### 4. 验证 (Verify)

- 运行全量测试套件，确认零失败
- 检查覆盖率报告，确认达到目标
- 验证测试在 CI 环境中可稳定运行
- 确认测试执行时间在合理范围内
- 变异测试（如适用）：确认测试能捕获代码变更
- 检查测试报告格式和可读性

→ **验证门禁**：测试全部通过 + 覆盖率 ≥ 目标 + CI 稳定 + 无 flaky test

### 5. 交付 (Deliver)

- 更新测试文档和运行说明
- PR 包含覆盖率变化报告
- 标注新增的测试依赖和环境要求
- 测试执行结果附在 PR 描述中
- 更新测试计划文档（如涉及）

→ **交付门禁**：测试实际运行并通过 + 覆盖率报告已附 + 文档已更新

## 自主执行策略

默认自主执行。仅在以下情况询问用户：
1. 需求有多种互斥解读，无法自行判断
2. 高风险/破坏性操作（重置测试数据库、修改 CI 配置）
3. 需要外部服务凭证或测试环境权限

反模式（明确禁止）：
- "要我继续吗？" → 默认继续
- "要我创建文件吗？" → 直接创建
- "要我运行测试吗？" → 直接运行
- "要我修复 flaky test 吗？" → 直接修复
- "这样可以吗？" → 做出判断，执行最佳方案

## 回退策略

```
Implementation 失败 → 回退到 Planning，重新评估方案
Verify 失败 → 回退到 Implementation，修复后重新验证
同一失败出现 2 次 → 换策略（不同测试框架/方法）
换策略后仍失败 → 上报用户，附带已尝试方案和失败原因
```

## 跨领域原则

- **测试金字塔**：单元 > 集成 > E2E，底层覆盖优先
- **测试即文档**：测试名称和结构应清晰表达被测行为
- **确定性优先**：flaky test 比无测试更有害
- **快速反馈**：单元测试 < 10s，集成测试 < 60s，全量 < 5min
- **测试隔离**：每个测试独立运行，无状态泄漏
- **数据驱动**：用参数化覆盖多场景，避免复制粘贴
- **持续改进**：追踪覆盖率趋势，识别覆盖盲区

## 评估敏感信号检测

当检测到以下信号时，提升验证严格度：
- 涉及支付/金融逻辑的测试
- 涉及认证/授权的测试
- 涉及数据迁移的验证
- 历史上频繁出 bug 的模块
- 性能关键路径的测试
