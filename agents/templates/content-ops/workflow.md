# 工作流规范

## 任务评估

收到任务后首先评估类型，选择对应路由：

| 任务类型 | 路由 | 触发条件 |
|---------|------|---------|
| Bug Fix | 调试路由 | 内容错误、断链、格式损坏、翻译偏差 |
| Feature | 功能路由 | 新文档、新内容系列、内容架构搭建 |
| Refactor | 重构路由 | 内容重组、术语统一、风格指南升级 |
| 简单问答 | 直接回复 | 不涉及内容变更的咨询 |

评估维度：
- 内容类型：技术文档 / 产品文案 / 帮助中心 / 博客 / 变更日志
- 目标受众：开发者 / 终端用户 / 管理员 / 潜在客户
- 影响范围：单篇 / 系列 / 全站

## 强制技能链（不可跳过）

### Feature 路由
```
planning → architecture(仅大型变更) → testing-strategy → change-validation → code-review
```

### Bug Fix 路由
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### Refactor 路由
```
planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix **必须**使用 sde-debug 和 testing-strategy，不可跳过
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review **始终**是最后一步，不可跳过
- testing-strategy 必须在 change-validation 之前完成

## 公共步骤

### 1. 理解 (Understand)

- 明确内容目标：教育 / 引导 / 告知 / 说服
- 确认目标受众和他们的知识水平
- 识别内容格式和渠道要求
- 收集素材：产品需求、技术方案、用户反馈
- 确认 SEO 要求和关键词策略

→ **理解门禁**：能准确复述内容目标 + 受众已明确 + 素材已收集

### 2. 调查 (Investigate)

- 查看现有相关内容，避免重复
- 确认术语表和风格指南要求
- 与 SME（Subject Matter Expert）沟通确认技术细节
- 分析竞品内容策略和质量标杆
- 检查内容分析数据（阅读量、跳出率、搜索排名）

调查方法（内容运营特定）：
- 内容 audit：现有内容的完整性和时效性
- 用户搜索词分析：用户在找什么
- 支持工单分析：用户遇到什么问题
- 竞品内容结构和覆盖范围对比

→ **调查门禁**：现有内容已审查 + 素材已就绪 + 风格指南已确认

### 3. 实现 (Implement)

- 按模板和风格指南撰写
- 结构化组织：概述 → 步骤 → 示例 → 常见问题
- 添加必要的代码示例、截图、图表
- 标注相关链接和前置条件
- 编写元数据：标题、描述、标签、分类
- 准备多语言版本（如需要）

实现规范：
- 单一来源原则：一个主题只在一个地方详述
- 渐进披露：从简单到复杂，让读者自行决定深度
- 代码示例必须可运行
- 截图必须有替代文本
- 链接必须指向具体章节，不指向页面顶部

→ **契约对齐门禁**：内容与规划一致 + 格式规范 + 链接有效

### 4. 验证 (Verify)

- 自查：拼写、语法、格式、链接可用性
- 技术审核：由 SME 验证准确性
- 受众审核：找目标受众试读反馈
- 代码示例验证：实际运行确认可用
- SEO 检查：标题、描述、关键词密度
- 无障碍检查：图片 alt 文本、阅读顺序

→ **验证门禁**：技术审核通过 + 无断链 + 代码示例可运行

### 5. 交付 (Deliver)

- 发布到对应渠道
- 设置定期巡检提醒
- 配置内容分析追踪
- 通知相关方（产品、支持、营销）
- 更新内容地图和索引

→ **交付门禁**：内容已发布 + 链接已验证 + 追踪已配置

## 自主执行策略

默认自主执行。仅在以下情况询问用户：
1. 需求有多种互斥解读，无法自行判断
2. 涉及品牌定位或法律合规敏感内容
3. 需要 SME 确认但无法联系

反模式（明确禁止）：
- "要我继续吗？" → 默认继续
- "要我创建文件吗？" → 直接创建
- "要我写文档吗？" → 直接编写
- "要我校对吗？" → 直接校对
- "这样可以吗？" → 做出判断，执行最佳方案

## 回退策略

```
Implementation 失败 → 回退到 Planning，重新评估方案
Verify 失败 → 回退到 Implementation，修复后重新验证
同一失败出现 2 次 → 换策略（不同内容结构/表述方式）
换策略后仍失败 → 上报用户，附带已尝试方案和失败原因
```

## 跨领域原则

- **单一来源**：一个主题只在一个地方详述，其他地方引用
- **渐进披露**：从简单到复杂，让读者自行决定深度
- **内容即产品**：用产品思维管理内容（用户反馈 → 迭代改进）
- **可维护性**：写的时候就想好谁来维护、怎么维护
- **可搜索性**：标题、标签、描述服务于用户搜索习惯
- **时效性管理**：设置内容过期机制，定期审查更新
- **本地化就绪**：文案结构适配翻译流程

## 评估敏感信号检测

当检测到以下信号时，提升验证严格度：
- 涉及法律/合规相关内容
- 涉及定价或付费功能描述
- 涉及安全相关操作指引
- 面向外部用户的公开内容
- 涉及 API 破坏性变更的文档
