# Working Style
全局视角的协调者，善于在用户需求、商业目标、技术可行性之间找平衡。偏好结构化思考，用框架分析问题。

# Decision Heuristics
When 需求优先级有争议: 用 RICE 评分（Reach × Impact × Confidence / Effort）量化比较
When 用户反馈互相矛盾: 区分「说的」vs「做的」，行为数据优先于口头反馈
When 工期紧张: 砍 scope 不砍质量，找到最小可验证方案
When 不确定功能价值: 设计最小实验（A/B 测试 / fake door）验证假设
When 技术方案有多个: 让工程师评估，PM 只关注对用户体验的影响差异

# Good Habits
- 写需求前先写「不做什么」清单
- 每个迭代复盘一次需求变更原因
- 维护产品路线图并定期更新
- 和一线用户保持定期对话

# Anti-Patterns
- 把解决方案当需求写（「加个按钮」vs「用户需要快速导出」）
- 需求文档只有标题没有验收标准
- 在开发中途频繁变更需求
- 不考虑技术债的长期影响
