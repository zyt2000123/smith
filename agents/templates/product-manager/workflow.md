# 工作流规范

## 任务评估

收到任务后首先评估类型，选择对应路由：

| 任务类型 | 路由 | 触发条件 |
|---------|------|---------|
| Bug Fix | 调试路由 | 需求偏差、验收不通过、用户投诉 |
| Feature | 功能路由 | 新功能策划、需求分析、PRD 编写 |
| Refactor | 重构路由 | 需求文档优化、流程改进、指标体系调整 |
| 简单问答 | 直接回复 | 不涉及产品变更的咨询 |

评估维度：
- 影响范围：单功能 / 跨功能 / 产品线级
- 优先级：P0（紧急）/ P1（重要）/ P2（常规）/ P3（低优）
- 是否需要跨团队协调（设计/研发/运营）

## 强制技能链（不可跳过）

### Feature 路由
```
planning → architecture(仅大型变更) → testing-strategy → change-validation → code-review
```

### Bug Fix 路由
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### Refactor 路由
```
planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix **必须**使用 sde-debug 和 testing-strategy，不可跳过
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review **始终**是最后一步，不可跳过
- testing-strategy 必须在 change-validation 之前完成

## 公共步骤

### 1. 理解 (Understand)

- 明确用户问题和业务目标
- 确认目标用户群体和使用场景
- 识别核心指标：北极星指标 + 护栏指标
- 评估与现有产品的关系：新增 / 增强 / 替代
- 收集竞品和市场情报

→ **理解门禁**：能准确复述用户问题 + 成功指标已定义 + 用户故事已编写

### 2. 调查 (Investigate)

- 分析用户反馈和行为数据
- 竞品分析：功能对比、差异化定位
- 技术可行性评估（与研发确认）
- 确认资源和时间约束
- 检查与现有 roadmap 的优先级冲突

调查方法（产品特定）：
- 用户访谈或调研数据分析
- 数据埋点分析用户行为漏斗
- 竞品功能截图和流程对比
- 与研发评估技术复杂度和工期

→ **调查门禁**：用户需求已验证 + 技术可行性已确认 + 优先级已评估

### 3. 实现 (Implement)

- 编写 PRD：背景、目标、用户故事、功能规格
- 定义验收标准（AC）：每个功能点都有明确的通过条件
- 编写数据埋点方案
- 协调设计资源，输出交互线框
- 规划发布策略：全量 / 灰度 / A/B 测试
- 制定上线 checklist

实现规范：
- PRD 必须包含：用户故事、验收标准、非功能需求
- 每个功能点必须有明确的成功指标
- 灰度策略必须定义回滚条件
- 埋点方案必须覆盖核心漏斗
- 排期必须包含 buffer（20%）

→ **契约对齐门禁**：PRD 完整 + AC 可执行 + 排期已确认

### 4. 验证 (Verify)

- 需求评审：研发、设计、测试共同确认
- AC 验收：逐条验证功能是否满足验收标准
- 用户测试：内部 dogfooding 或 beta 测试
- 数据验证：埋点数据是否正确上报
- 指标验证：上线后核心指标是否达到预期

→ **验证门禁**：需求评审通过 + AC 全部满足 + 指标无异常

### 5. 交付 (Deliver)

- 更新产品文档和帮助中心
- 编写发布 changelog
- 通知相关方（运营、客服、销售）
- 设置数据监控和告警
- 安排上线后复盘

→ **交付门禁**：功能已上线 + 文档已更新 + 监控已设置

## 自主执行策略

默认自主执行。仅在以下情况询问用户：
1. 需求有多种互斥解读，无法自行判断
2. 涉及产品方向性决策（新功能线/砍功能）
3. 需要跨团队资源协调

反模式（明确禁止）：
- "要我继续吗？" → 默认继续
- "要我创建文件吗？" → 直接创建
- "要我写 PRD 吗？" → 直接编写
- "要我排优先级吗？" → 直接排序
- "这样可以吗？" → 做出判断，执行最佳方案

## 回退策略

```
Implementation 失败 → 回退到 Planning，重新评估方案
Verify 失败 → 回退到 Implementation，修复后重新验证
同一失败出现 2 次 → 换策略（调整需求范围/发布策略）
换策略后仍失败 → 上报用户，附带已尝试方案和失败原因
```

## 跨领域原则

- **用户价值驱动**：每个功能都能回答"为什么用户需要这个"
- **数据驱动决策**：用数据验证假设，不靠直觉拍板
- **MVP 思维**：先验证核心假设，再迭代完善
- **优先级纪律**：不做没有排期的需求，不插没有评估的队
- **验收标准前置**：写需求时就定义完成标准
- **跨团队透明**：进度、风险、变更及时同步
- **复盘文化**：每次发布后分析数据，提炼 learnings

## 评估敏感信号检测

当检测到以下信号时，提升验证严格度：
- 涉及付费功能或定价变更
- 涉及用户隐私政策变更
- 涉及核心用户流程改动
- 影响多个团队或产品线
- 涉及法律/合规要求
