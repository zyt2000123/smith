# 工作流规范

## 任务评估

收到任务后首先评估类型，选择对应路由：

| 任务类型 | 路由 | 触发条件 |
|---------|------|---------|
| Bug Fix | 调试路由 | 用户报告异常、错误复现、回归问题 |
| Feature | 功能路由 | 新功能、增强、组件开发 |
| Refactor | 重构路由 | 代码优化、架构调整、技术债清理 |
| 简单问答 | 直接回复 | 不涉及代码变更的咨询 |

评估维度：
- 影响范围：仅限单组件 / 跨页面 / 影响设计系统
- 风险等级：低（样式调整）/ 中（逻辑变更）/ 高（架构改动）
- 是否需要设计稿或 API 文档确认

## 强制技能链（不可跳过）

### Feature 路由
```
planning → architecture(仅大型变更) → testing-strategy → change-validation → code-review
```

### Bug Fix 路由
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### Refactor 路由
```
planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix **必须**使用 sde-debug 和 testing-strategy，不可跳过
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review **始终**是最后一步，不可跳过
- testing-strategy 必须在 change-validation 之前完成

## 公共步骤

### 1. 理解 (Understand)

- 通读设计稿/需求，标注不清晰的交互和边界情况
- 确认 API 接口是否就绪，字段是否对齐
- 列出需要的组件依赖和数据流
- 确认浏览器兼容性和响应式断点要求
- 识别无障碍需求（ARIA 角色、键盘导航）

→ **理解门禁**：能准确复述需求 + 识别至少 3 个边界条件 + 明确技术约束

### 2. 调查 (Investigate)

- 检查现有组件库，确认是否有可复用组件
- 在浏览器 DevTools 中检查当前页面结构和样式
- 确认框架版本约束和浏览器兼容矩阵
- 如涉及新技术方案，做 spike 验证可行性
- 检查相关组件的 Storybook 文档

调查方法（前端特定）：
- 使用 DevTools Elements 面板检查 DOM 结构
- 使用 Network 面板确认 API 调用模式
- 使用 Performance 面板识别渲染瓶颈
- 使用 Lighthouse 建立性能基线

→ **调查门禁**：已确认可复用资源 + 技术方案可行 + 依赖已明确

### 3. 实现 (Implement)

- 先搭建组件骨架和 TypeScript 类型定义
- 实现核心逻辑，用 mock 数据跑通主流程
- 补充边界处理：loading / error / empty / 极端数据
- 实现响应式布局，覆盖移动端到桌面端
- 添加 ARIA 属性和键盘交互支持
- 对接真实 API，处理数据转换和错误状态
- 编写单元测试覆盖核心路径

实现规范：
- 组件 props 必须有 TypeScript 类型定义
- CSS 使用项目约定方案（CSS Modules / Tailwind / styled-components）
- 不硬编码文案，使用 i18n key
- 每个路由级组件包裹 ErrorBoundary

→ **契约对齐门禁**：实现方案与规划一致 + 类型完整 + lint 零错误

### 4. 验证 (Verify)

- 视觉比对：设计稿 vs 实际渲染，逐像素核查
- 交互验证：键盘导航、焦点管理、表单校验反馈
- 响应式测试：至少覆盖 mobile (375px) / tablet (768px) / desktop (1280px)
- 性能检查：Lighthouse 跑分、Bundle 大小分析
- 无障碍审查：axe 扫描零违规
- 跨浏览器测试：Chrome + Firefox + Safari（如项目要求）
- 运行完整测试套件，确认无回归

→ **验证门禁**：测试全部通过 + Lighthouse 性能分 ≥ 90 + 无回归 + 无障碍零违规

### 5. 交付 (Deliver)

- PR 附设计稿对比截图或录屏
- 更新 Storybook 文档（如涉及组件变更）
- 更新组件 API 文档
- 测试结果附在 PR 描述中
- 标注破坏性变更（如有）

→ **交付门禁**：测试实际运行并通过 + 截图/录屏已附 + 文档已更新

## 自主执行策略

默认自主执行。仅在以下情况询问用户：
1. 需求有多种互斥解读，无法自行判断
2. 高风险/破坏性操作（删除组件、修改公共 API）
3. 需要选择影响用户体验的设计方案

反模式（明确禁止）：
- "要我继续吗？" → 默认继续
- "要我创建文件吗？" → 直接创建
- "要我运行测试吗？" → 直接运行
- "要我安装依赖吗？" → 直接安装
- "这样可以吗？" → 做出判断，执行最佳方案

## 回退策略

```
Implementation 失败 → 回退到 Planning，重新评估方案
Verify 失败 → 回退到 Implementation，修复后重新验证
同一失败出现 2 次 → 换策略（不同技术方案/架构）
换策略后仍失败 → 上报用户，附带已尝试方案和失败原因
```

## 跨领域原则

- **渐进增强**：核心功能不依赖 JS，CSS 优先
- **无障碍优先**：ARIA 属性不是事后补丁，从设计阶段就纳入
- **国际化就绪**：不硬编码文案，使用 i18n 框架
- **错误边界**：每个路由级组件包裹 ErrorBoundary
- **性能预算**：首屏 LCP < 2.5s，交互 INP < 200ms
- **组件化思维**：单一职责，可组合，可测试
- **设计系统一致性**：优先使用设计系统 token，不自定义颜色/间距

## 评估敏感信号检测

当检测到以下信号时，提升验证严格度：
- 涉及支付/金融相关 UI
- 涉及用户隐私数据展示
- 涉及无障碍合规要求
- 性能关键路径（首屏渲染、大列表）
- 设计系统公共组件变更
