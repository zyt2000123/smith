# 工作流规范

## 任务评估

先判断任务属于哪一类，再选择最短完成路径：

| 任务类型 | 路由 | 触发条件 |
|---|---|---|
| Research | 调研路由 | 需要理解现状、查资料、梳理方案 |
| Execute | 执行路由 | 需要改代码、改配置、写文档、跑命令 |
| Review | 审查路由 | 需要发现风险、评估质量、给出判断 |
| Organize | 整理路由 | 需要总结、拆任务、归纳结论、形成行动清单 |

评估维度：
- 是否要落文件或运行命令
- 是否存在环境/权限/数据风险
- 是否依赖当前仓库或本地上下文
- 交付物是结论、改动、脚本还是操作步骤

## 强制技能链（按需触发）

### 执行类任务
```
planning → testing-strategy → change-validation → code-review
```

### Bug Fix / 排障
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### 调研 / 整理
```
clarify-goal → gather-evidence → synthesize → deliver
```

硬规则：
- 涉及代码或配置修改时，先确认现状再动手
- 涉及风险操作时，必须先显式提醒影响范围
- 能验证的结果要验证，不能只停在“理论上可行”
- 输出必须回到用户目标，而不是只汇报过程

## 公共步骤

### 1. 对齐目标 (Align)

- 提炼用户真正要完成的事
- 识别约束：时间、范围、风险、是否允许改代码
- 确认交付物：答案、改动、脚本、命令、文档

→ **对齐门禁**：能用一句话复述目标 + 知道什么算完成

### 2. 收集上下文 (Gather)

- 读取当前仓库、文件、配置、运行状态
- 检查是否已有现成实现或历史约定
- 需要外部信息时先搜索，再抓取原文

→ **上下文门禁**：关键事实已有证据，不靠猜测推进

### 3. 推进执行 (Advance)

- 选择最小可行路径
- 在必要时拆步，但不要把简单事过度流程化
- 改动尽量局部，保持和现有结构一致

→ **执行门禁**：方案能落地 + 影响面清楚 + 未越过架构边界

### 4. 验证结果 (Verify)

- 跑对应测试、命令或最小人工验证
- 记录失败点、限制和剩余风险
- 必要时回退到前一步调整方案

→ **验证门禁**：结果被证实，而不是只被描述

### 5. 清晰交付 (Deliver)

- 先给结果，再给关键证据
- 说明改了什么、没改什么、为什么
- 给出明确的下一步或使用方式

→ **交付门禁**：用户能立即继续工作，不需要二次猜测

## 自主执行策略

默认自主推进。仅在以下情况停下来确认：
1. 需求存在互斥理解，且代价明显不同
2. 会触发高风险或不可逆副作用
3. 缺少继续执行所必须的外部权限或输入

## 回退策略

```
收集证据不足 → 回到 Gather
执行验证失败 → 回到 Advance
同一路径连续失败 2 次 → 换一种更小、更稳的方案
仍无法推进 → 向用户报告阻塞点和已验证事实
```
