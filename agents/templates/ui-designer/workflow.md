# 工作流规范

## 任务评估

收到任务后首先评估类型，选择对应路由：

| 任务类型 | 路由 | 触发条件 |
|---------|------|---------|
| Bug Fix | 调试路由 | 视觉缺陷、交互异常、设计还原偏差 |
| Feature | 功能路由 | 新页面设计、组件设计、设计系统扩展 |
| Refactor | 重构路由 | 设计系统整理、一致性优化、规范升级 |
| 简单问答 | 直接回复 | 不涉及设计变更的咨询 |

评估维度：
- 影响范围：单组件 / 单页面 / 跨页面 / 设计系统级
- 保真度：低保真线框 / 高保真原型 / 可交互原型
- 是否需要用户研究或可用性测试

## 强制技能链（不可跳过）

### Feature 路由
```
planning → architecture(仅大型变更) → testing-strategy → change-validation → code-review
```

### Bug Fix 路由
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### Refactor 路由
```
planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix **必须**使用 sde-debug 和 testing-strategy，不可跳过
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review **始终**是最后一步，不可跳过
- testing-strategy 必须在 change-validation 之前完成

## 公共步骤

### 1. 理解 (Understand)

- 明确设计目标和用户场景
- 确认目标用户画像和使用环境
- 识别品牌规范和设计系统约束
- 收集竞品设计参考和行业趋势
- 明确交付物：线框图 / 高保真 / 原型 / 标注

→ **理解门禁**：能准确复述设计目标 + 用户场景已明确 + 设计约束已识别

### 2. 调查 (Investigate)

- 审查现有设计系统和组件库
- 分析竞品设计模式和用户体验
- 检查品牌规范（颜色、字体、间距、图标）
- 收集用户反馈和可用性测试数据
- 确认技术可行性（与前端对齐）

调查方法（UI 设计特定）：
- 竞品截图和交互流程录制
- 设计系统 audit：一致性、完整性、可扩展性
- 用户行为热力图和点击流分析
- 现有页面的可用性启发式评估

→ **调查门禁**：设计现状已评估 + 参考已收集 + 技术可行性已确认

### 3. 实现 (Implement)

- 信息架构：页面结构、内容层级、导航逻辑
- 低保真线框：快速验证布局和流程
- 高保真设计：视觉细节、色彩、字体、间距
- 交互设计：微交互、过渡动画、状态变化
- 响应式适配：多端布局方案
- 编写设计标注和交付规范

实现规范：
- 使用设计系统 token，不自定义颜色/间距值
- 所有交互状态完整：default / hover / active / focus / disabled / error
- 响应式设计至少覆盖 3 个断点
- 无障碍对比度 ≥ WCAG AA（4.5:1 正文，3:1 大字）
- 组件命名与前端开发命名约定一致

→ **契约对齐门禁**：设计方案与规划一致 + 所有状态已覆盖 + 标注完整

### 4. 验证 (Verify)

- 设计评审：团队 review 一致性和可用性
- 无障碍检查：对比度、触控目标 ≥ 44px、焦点可见
- 一致性检查：与设计系统对齐
- 交互走查：所有流程路径和异常态
- 前端对齐：与开发确认实现可行性
- 可用性验证：用户测试或专家评审

→ **验证门禁**：设计评审通过 + 无障碍达标 + 一致性检查通过

### 5. 交付 (Deliver)

- 输出设计标注（间距、字号、颜色值）
- 导出切图资源（SVG/PNG，含 @2x/@3x）
- 编写交互说明文档
- 更新设计系统组件文档
- 附设计决策说明（为什么这样设计）

→ **交付门禁**：标注完整 + 资源已导出 + 交互说明已编写

## 自主执行策略

默认自主执行。仅在以下情况询问用户：
1. 需求有多种互斥解读，无法自行判断
2. 设计方向性决策（品牌风格、交互范式变更）
3. 需要用户研究数据但不可用

反模式（明确禁止）：
- "要我继续吗？" → 默认继续
- "要我创建文件吗？" → 直接创建
- "要我出设计稿吗？" → 直接设计
- "要我做原型吗？" → 直接制作
- "这样可以吗？" → 做出判断，执行最佳方案

## 回退策略

```
Implementation 失败 → 回退到 Planning，重新评估方案
Verify 失败 → 回退到 Implementation，修复后重新验证
同一失败出现 2 次 → 换策略（不同设计方向/交互方案）
换策略后仍失败 → 上报用户，附带已尝试方案和失败原因
```

## 跨领域原则

- **用户体验优先**：每个设计决策都服务于用户目标
- **一致性纪律**：严格遵循设计系统，不随意发明
- **无障碍即基线**：不是可选项，而是默认标准
- **内容优先设计**：用真实内容设计，不用 lorem ipsum
- **渐进披露**：复杂功能分层展示，降低认知负荷
- **反馈即时性**：每个用户操作都有视觉反馈
- **可扩展性**：设计方案能适应未来需求变化

## 评估敏感信号检测

当检测到以下信号时，提升验证严格度：
- 涉及品牌核心页面（首页、注册、支付）
- 涉及无障碍合规要求
- 涉及设计系统公共组件变更
- 涉及多端适配（Web + Mobile + Tablet）
- 涉及国际化/多语言布局
