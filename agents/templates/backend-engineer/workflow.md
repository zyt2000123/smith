# 工作流规范

## 任务评估

收到任务后首先评估类型，选择对应路由：

| 任务类型 | 路由 | 触发条件 |
|---------|------|---------|
| Bug Fix | 调试路由 | 异常日志、错误响应、数据不一致 |
| Feature | 功能路由 | 新 API、数据模型变更、服务集成 |
| Refactor | 重构路由 | 性能优化、架构调整、技术债清理 |
| 简单问答 | 直接回复 | 不涉及代码变更的咨询 |

评估维度：
- 影响范围：单服务 / 跨服务 / 涉及数据迁移
- 风险等级：低（只读查询）/ 中（写操作）/ 高（数据迁移/破坏性变更）
- 是否需要数据库迁移或外部服务协调

## 强制技能链（不可跳过）

### Feature 路由
```
planning → architecture(仅大型变更) → testing-strategy → change-validation → code-review
```

### Bug Fix 路由
```
sde-debug → planning → testing-strategy → change-validation → code-review
```

### Refactor 路由
```
planning → testing-strategy → change-validation → code-review
```

硬规则：
- Bug Fix **必须**使用 sde-debug 和 testing-strategy，不可跳过
- architecture 仅在变更涉及 3+ 文件或跨模块时触发
- code-review **始终**是最后一步，不可跳过
- testing-strategy 必须在 change-validation 之前完成

## 公共步骤

### 1. 理解 (Understand)

- 明确输入输出的数据结构和约束
- 确认权限模型：谁能调用、什么条件下可调用
- 识别依赖：数据库、缓存、消息队列、外部服务
- 确认数据一致性要求（强一致 / 最终一致）
- 识别并发场景和竞态条件风险

→ **理解门禁**：能准确复述需求 + 识别至少 3 个边界条件 + 明确数据流向

### 2. 调查 (Investigate)

- 审查现有数据模型和 API 是否可复用
- 检查是否有相似的业务逻辑已实现
- 确认数据库索引和查询模式
- 检查日志系统，确认可观测性覆盖
- 审查现有中间件和拦截器

调查方法（后端特定）：
- 检查数据库查询计划（EXPLAIN ANALYZE）
- 审查应用日志和错误追踪（Sentry/ELK）
- 检查 API 文档和接口契约
- 分析慢查询日志和性能指标

→ **调查门禁**：已确认可复用资源 + 技术方案可行 + 依赖已明确

### 3. 实现 (Implement)

- 先定义接口契约（请求/响应 schema，含错误码）
- 实现数据层：模型定义、数据库迁移、仓储接口
- 实现服务层：业务逻辑、输入校验、错误处理
- 实现接口层：路由、中间件、序列化/反序列化
- 编写单元测试和集成测试
- 添加结构化日志和指标埋点

实现规范：
- 分层清晰：Controller → Service → Repository，不跨层调用
- 配置外部化：环境变量或配置中心，不硬编码
- 幂等性：POST 之外的操作天然幂等，POST 设计重试安全机制
- API 版本化：URL 或 Header 方式，不破坏现有消费者
- 数据库迁移必须可回滚

→ **契约对齐门禁**：实现方案与规划一致 + 接口契约完整 + lint 零错误

### 4. 验证 (Verify)

- 运行全量测试套件（单元 + 集成 + 契约测试）
- 检查 SQL 查询计划，确认无全表扫描
- 用 curl/httpie 手动验证关键路径
- 检查错误响应格式是否统一
- 验证数据库迁移的正向和回滚
- 执行基本负载测试（如涉及高频接口）
- 检查日志输出是否符合结构化格式

→ **验证门禁**：测试全部通过 + 无 N+1 查询 + 无回归 + 迁移可回滚

### 5. 交付 (Deliver)

- 更新 API 文档（OpenAPI/Swagger）
- 编写数据库迁移的回滚步骤
- PR 包含测试覆盖率报告
- 标注破坏性变更和迁移步骤
- 更新 CHANGELOG（如涉及公共 API 变更）

→ **交付门禁**：测试实际运行并通过 + API 文档已更新 + 迁移回滚已验证

## 自主执行策略

默认自主执行。仅在以下情况询问用户：
1. 需求有多种互斥解读，无法自行判断
2. 高风险/破坏性操作（数据迁移、删表、修改公共 API）
3. 需要外部服务权限或凭证

反模式（明确禁止）：
- "要我继续吗？" → 默认继续
- "要我创建文件吗？" → 直接创建
- "要我运行测试吗？" → 直接运行
- "要我执行迁移吗？" → 直接执行（开发环境）
- "这样可以吗？" → 做出判断，执行最佳方案

## 回退策略

```
Implementation 失败 → 回退到 Planning，重新评估方案
Verify 失败 → 回退到 Implementation，修复后重新验证
同一失败出现 2 次 → 换策略（不同技术方案/架构）
换策略后仍失败 → 上报用户，附带已尝试方案和失败原因
```

## 跨领域原则

- **安全优先**：输入校验、参数化查询、最小权限原则
- **数据一致性**：事务边界清晰，分布式场景用 Saga/补偿
- **可观测性**：结构化日志 + 指标 + 追踪，三支柱齐全
- **幂等设计**：所有写操作都应设计为可安全重试
- **优雅降级**：外部依赖不可用时有 fallback 方案
- **配置外部化**：零硬编码，环境变量或配置中心
- **版本管理**：API 版本化 + 数据库迁移有序号

## 评估敏感信号检测

当检测到以下信号时，提升验证严格度：
- 涉及支付/金融数据处理
- 涉及用户隐私数据（PII/PHI）
- 涉及认证/授权逻辑变更
- 涉及数据库 schema 变更
- 涉及跨服务通信协议变更
