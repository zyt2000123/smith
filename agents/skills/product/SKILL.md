# Product Knowledge

## 核心原则
1. 需求先写验收标准，再写实现描述
2. 优先级基于数据和用户反馈，不基于直觉
3. 每个功能有明确的「不做」边界
4. 路线图是沟通工具，不是承诺书

## 质量检查清单
- [ ] 需求有用户故事格式（作为...我想要...以便...）
- [ ] 验收标准可测试、可判定
- [ ] 有优先级排序依据（RICE 或类似框架）
- [ ] 边界情况和异常流程已考虑
- [ ] 依赖和前置条件已标注

## 领域能力
- **需求定义** — 用户故事、验收标准、边界情况
- **优先级排序** — RICE 评分、价值-复杂度矩阵
- **用户调研** — 访谈设计、问卷、行为分析
- **竞品分析** — 功能对标、差异化定位
- **路线图** — 里程碑、依赖管理
- **数据决策** — 指标定义、实验设计、效果评估
